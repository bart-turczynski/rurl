# P4.1 — Scheme specialization posture

## Relationship to P2.4

P2.4 (`decisions/P2.4-scheme-admission-boundary.md`, same branch) settles the
**admission** half of contradiction **C-11**: *which* scheme tokens may enter
parsing under which axis setting. It defines the two named admission classes
`ADMIT-WEB` / `ADMIT-GENERAL`, the admission predicate, and the six-point
interface (its **§D-D**) that this record must honor. **P4.1 settles the
specialization half of C-11**: *what the parser does with each admitted,
specialized scheme.* Joint disposition of C-11 = **P2.4 (admission) + P4.1
(specialization)**; neither closes C-11 alone (see G2.A below).

This record **consumes P2.4's admission lattice unchanged**. It does **not**
re-admit a token that `web` rejects, does **not** reject a token that `general`
admits, and does **not** widen the default `web` allowlist. Every posture below
operates strictly *within* the set P2.4 admits. Concretely, per **P2.4 §D-D**:

- **§D-D.1 (admission predicate fixed).** P4.1 specializes only tokens satisfying
  `(web ∧ lower(token) ∈ .SUPPORTED_SCHEMES) ∨ (general ∧ token matches
  ^[A-Za-z][A-Za-z0-9+.-]*$ ∧ url_standard ≠ NULL)`.
- **§D-D.2 (admission ≠ success).** P4.1 owns the per-scheme *fatal-vs-reportable*
  boundaries for the admitted set — the boundaries P2.4 explicitly delegates.
- **§D-D.3 (special-ness is interpretation, not admission).** P4.1 owns the
  special/non-special model; it is a fact of `url_standard`, never of admission.
- **§D-D.4 (`web` allowlist is compatibility-frozen).** P4.1 treats no non-`web`
  scheme as admissible under `web`; any default-allowlist widening is a versioned
  P-tier decision, never a P4.1 specialization.
- **§D-D.5 (ws/wss inert under `web`).** P4.1 specializes ws/wss **only** under
  `general`.
- **§D-D.6 (mailto: PSL carve-out is specialization, not admission widening).**
  P4.1 defines the `mailto:` recipient projection and its PSL carve-out without
  altering the admission predicate.

P4.1 records how each admitted scheme is interpreted/specialized. The posture is
**preservation of shipped ADR 0012 behavior**, made normative per-family, with
every specialization gated on `url_standard` (interpretation) and never on
`scheme_acceptance` (admission).

### D-A. Special vs non-special scheme model (item 1)

**Special-ness is a WHATWG-interpretation fact, owned by `url_standard`, never an
admission fact** (P2.4 §D-D.3; ADR 0012 A.1 `#special-scheme`).

1. **Under `url_standard = "whatwg"`**, exactly six schemes are **special** —
   `http`, `https`, `ws`, `wss`, `ftp`, `file` — verified against the shipped
   `.WHATWG_SPECIAL_SCHEMES` (`R/utils.R:38` =
   `c("http","https","ftp","ws","wss","file")`). Special schemes get authority +
   host + the WHATWG host model + default-port handling; a **special scheme with
   a missing/empty host is a host-missing parse `error`** (ADR 0012 D2, D3
   fatal-vs-reportable). Every **other** admitted scheme is **non-special** and
   parses as one of ADR 0012 D1's three shapes: opaque-path (`foo:bar`),
   non-special list-path without authority (`foo:/bar`), or non-special authority
   (`foo://host`, `foo:///bar` empty-host, `foo://[::1]/bar`).

2. **Under `url_standard = "rfc3986"` (`rfc-syntax` posture)** there is **no
   special/non-special classification at all** (ADR 0012 D1: "RFC 3986 has no
   corresponding classification"). An admitted scheme parses against RFC 3986's
   generic `URI` grammar via D1's *independent* generic-grammar gate (ADR 0012 D1
   "Deterministic RFC-general failure boundary"; Layer 4a), not delegated to
   libcurl permissiveness. The default-port convention is available to any scheme
   that declares one (RFC 3986 §3.2.3), which is an interpretation fact, not a
   special-ness fact.

3. **`get_scheme_class()`** surfaces `special` / `non-special` / opaque as a
   **companion diagnostic** (ADR 0006; public-surface-inventory row 81), fed by
   `get_scheme()` returning the literal scheme for accepted opaque input (ADR 0012
   D2 cascade). It is metadata, never a parse gate.

4. **Default ports are interpretation/normalization data, not admission and not a
   specialization license.** Shipped `.SCHEME_DEFAULT_PORTS` (`R/utils.R:104`) =
   `http=80, https=443, ftp=21, ws=80, wss=443` — deliberately **no `ftps`/`sftp`
   entry** (D-B). Default-port stripping is eligible for the scheme that owns a
   port; absence of a default port is itself the posture for ftps/sftp.

### D-B. FTP-family and scheme-relative posture (item 2)

**FTP-family posture (references P2.4 §D-A.3 and ADR 0012 Open-Q2 / RURL-lvrfrkfa,
*resolved*; does not re-decide admission):**

| Scheme | Admission (P2.4, fixed) | Specialization (this record) |
|---|---|---|
| `ftp` | `web` allowlist + `general` | WHATWG-**special**; host model + **default port 21**; DNS/PSL-eligible host (host detail → P4 host record); **no** HTTP(S) semantic transforms (ADR 0012 D2 — SEO transforms are HTTP(S)-only). |
| `ftps` | grandfathered in `web` allowlist + `general` (P2.4 §D-A.3) | **Non-special under WHATWG** (absent from `.WHATWG_SPECIAL_SCHEMES`), routed to the in-tree general parser; kept on the legacy curl-backed RFC route; **no standards-backed default port** (absent from `.SCHEME_DEFAULT_PORTS` by design — ADR 0012 D5, Open-Q2). Grandfathered for CRAN byte-compat, **not** standards backing. The `ftp` vs `ftps` difference is load-bearing (S6 Blocker 1). |
| `sftp` | `general`-only (never in `web` allowlist; P2.4 §D-A, ADR 0012 Open-Q2) | Non-special general parsing (WHATWG opaque/authority or RFC generic); **no default-port rule**; **no browser-fixer inference** (fixer PRD — unknown schemes are not rewritten). |

Any default port asserted for ftps/sftp would be convention, flagged as such,
**never** presented as standards-backed (ADR 0012 D5 de-facto-scheme clause).

**Scheme-relative `//host` posture.** Scheme-relative input is a **separate input
axis** — `scheme_relative_handling ∈ {keep, http, https, error}` (ADR 0010;
`R/parse.R:447`), **independent of `scheme_acceptance` and `scheme_policy`**
(S6 Medium 8; ADR 0012 D3 note). It is **not** a scheme-membership row and P4.1
does not fold it into the scheme model. Its per-profile defaults are recorded as
interpretation coordination, not re-decided here: `browser → http`, `whatwg →
error` (no invented base), `rfc-syntax → keep` (ADR 0012 D6). Whether the
`whatwg`-profile `scheme_policy="require"` vs direct-selector `infer` divergence
is tightened is the standards/profile-precedence slice's call (S6 O2; P2.4 Open
Question 5), **not** P4.1's.

### D-C. Generic credentials, mailto raw/decoded state, and email diagnostics (item 3)

**Generic (non-web) credentials.** Userinfo carried by a non-special authority
(`foo://u:p@host/a`, `ws://u:p@host/a`) is **authority credential state**, a
distinct route from scheme-less `user@host` (`warning-userinfo`), RFC `file:`
userinfo, and `mailto:` local parts (S6 Blocker 3; five distinct credential
routes). The **proposed posture is to preserve parsed generic authority
credentials** so that `user`/`password`/`userinfo` are populated and the row is
reassemblable, closing the shipped gap where `.parse_opaque_url_one()` calls
`.split_authority()` but **drops the returned userinfo** (S6 Blocker 3 evidence;
`R/parse-state.R`). Two items are **flagged for the owner, not decided here**:
(a) the exact clean-vs-exact serialization + security policy for
credential-bearing output (S6 O3/O9 — coordinate with the P2 serialization
decision), and (b) `get_password()` selector parity with `get_user()`/
`get_userinfo()` (S6 O4 — `get_password()` currently lacks the general selectors;
public-surface-inventory). See Open Questions 1–2.

**`mailto:` recipient raw/decoded state + PSL carve-out (P2.4 §D-D.6; ADR 0012
D7).** Under `general` acceptance a `mailto:` URL's **first** recipient decomposes
through the *same* accessors a web host uses — `get_host` / `get_domain` /
`get_tld` / `get_subdomain` / `get_user` / `get_userinfo` — via the **same PSL
seam** (ADR 0012 D7). This is the **one place a general-routed host reaches the
PSL seam**, a deliberate carve-out of ADR 0012 D2's general-host PSL mask, because
a recipient RHS *is* a domain. Posture:

- **Extraction metadata only.** The recipient domain populates internal Stage-A
  `host`; Stage-B still serializes from its own `gen_b` re-parse (host `NA` for
  mailto), so `clean_url` / round-trip is byte-for-byte unchanged (ADR 0012 D7).
- **Domain-form RHS only.** An address-literal or invalid RHS → `NA` host; the
  local part still resolves as `user`.
- **First recipient is a convenience selection, not a spec claim;** the full
  per-recipient breakdown is `get_mailto_recipients()`'s job.
- **Gated to `general`.** Under default `web`, `mailto:` does not parse, so every
  default-mode result is unchanged (P2.4 §D-D.6; ADR 0012 D7).
- **Raw-vs-decoded (S6 O5) is a genuine open decision, not settled here.** The
  shipped `.mailto_first_recipient_parts()` percent-decodes the first local part
  while `get_user()`'s doc claims raw/exact output — a documented mismatch (S6
  O5). P4.1 proposes to **align on decode-exactly-once with corrected docs**, but
  flags it for the owner (Open Question 3); the recipient-*helper* provenance
  (tokenize-before-decode, `%2C` non-delimiter) lives in the email PRD and the
  P4-email fixture work, not this record.

**Email-diagnostics surface.** `get_mailto_recipients()` (ADR 0012 D7;
`R/email-diagnostics.R:491`; public-surface-inventory row 74; defaults
`url_standard="rfc3986"`, `scheme_acceptance="general"`, `smtp_wire=FALSE`) is the
**only** email-specific helper; per ADR 0012 D7 and the [[rurl-email-mailto-d7]]
posture, rurl mints **no** `mailto`/email-specific extraction functions — it
reuses the web accessors. All email/SMTP *facts* are companion-only (ADR 0006);
the `smtp_wire` tier is opt-in and **must never alter URL acceptance** (S6 dial
inventory). Two accepted-vs-shipped email gaps are **referenced, not resolved
here**: the `indeterminate` unresolved-lexer contract (S6 O6 → D-D below) and the
URL-level email facts `userinfo-form` / `public-suffix-known` /
`smtp-envelope-wire-mode` (S6 O7). Their disposition is the P4-email slice's,
paired with the email PRD.

### D-D. Unresolved lexer contexts (item 4)

**Opaque-path / non-special path & authority lexing.** For non-special schemes
under `whatwg` (`tel:`, `data:`, arbitrary `foo:`), the parse shape is fixed by
ADR 0012 D1's opaque-path trigger (`#opaque-path-state`): after `:`, a URL goes
opaque **iff** the scheme is non-special AND the remainder does not start with
`/`. `foo:bar` → opaque path (path is a *string*); `foo:/bar` → non-special
list-path (host null); `foo://host/a` → non-special authority. Under `rfc-syntax`
the same bodies are `path-rootless` / `path-absolute` / authority with no
opaque/list discriminator (ADR 0012 D1 table). **Scheme-specific violations are
selected companion facts, never parse failures** (ADR 0012 D5 fatal-vs-reportable;
P2.4 §D-D.2): `data:` missing its mandatory comma, `tel:` missing
`;phone-context=`, `mailto:` carrying a fragment are each an `ok`-family parse
plus a diagnostic — not an `error`. A generic-grammar violation (RFC repeated
raw `@`, malformed `%`-triplet) *is* fatal under `rfc-syntax` (ADR 0012 D1
boundary).

**ws/wss are specialized only under `general` (P2.4 §D-D.5).** Under `general` +
`whatwg` they parse as WHATWG-**special** URLs with the full host model and
default ports 80/443, carrying selected fragment/userinfo facts (RFC 6455 forbids
both — reported, not enforced; ADR 0012 D5). Under `web` no ws/wss row survives
admission (the metadata in `.WHATWG_SPECIAL_SCHEMES` is admission-inert;
`R/utils.R:44,98`), so P4.1 defines **no** `web`-path ws/wss behavior.

**Indeterminate email lexer state (S6 O6) is referenced, not resolved here.** The
accepted email PRD requires an `indeterminate` outcome (unterminated quote /
domain-literal) that invents no recipients; the shipped splitter does not expose
it. That is a specialization detail of the P4-email slice + email PRD, outside
this record's single-file scope; P4.1 records only that no admitted `mailto:`
token may be turned into a parse `error` by an unresolved recipient boundary
(P2.4 §D-D.2).

### D-E. Host-vocabulary scope boundary (RCON-08)

Where a specialization forces a host posture (`file:` host/localhost/drive,
special-scheme mandatory host, ftp DNS/PSL-eligible host, mailto recipient
domain), P4.1 states **the posture** (special scheme ⇒ host mandatory; `file`
maps `file://localhost/…` to empty host per ADR 0012 D5; recipient RHS is
domain-decomposed) and **defers the host vocabulary detail** — IDNA profile/order,
PSL source identity and private-section policy, DNS-boundary, IP identity — to the
separate **P4 host record** (RCON-08; reconciliation §8 P4 "Host vocabulary, IDNA
profile, PSL policy/provenance, DNS boundary, and IP source/identity behavior").
This record does not decide RCON-08.

- **C-11 required disposition, verbatim** (reconciliation §5 line 334;

  precisely and retain the old rule only where compatibility/default policy still
  calls for it."* P2.4 supplied the *admission* residue; P4.1 supplies the

  interpretation, completing C-11's disposition.
- **RCON-05 open decisions** (reconciliation §4 lines 195-198): "`ftp`/`ftps`/
  `sftp` treatment and default ports" (D-B), "generic credentials" (D-C),
  "scheme-relative inputs" (D-B), "`mailto:` parsing and diagnostics" (D-C/D-D).
  D-B/D-C/D-D dispose of the *specialization* items in that list; the admission
  items ("v3 default admission", "compatibility aliases") are P2.4's.
- **RCON-08 boundary** (reconciliation §4 lines 247-268): the host/IDNA/PSL/DNS/IP
  typed contracts are explicitly a separate matrix. D-E honors that boundary,
  stating scheme-forced host posture only and deferring vocabulary.
- **ADR 0012 owns the specialization primitives** (D1 special/non-special + RFC
  generic gate; D2 HTTP(S)-only transforms + host/PSL eligibility; D5 selected
  facts + de-facto ports; D7 mailto recipient projection + PSL carve-out; Open-Q2
  ftps/sftp *resolved*). P4.1 makes these normative per-family without reopening
  them.
- **§8 P4 scope** (reconciliation lines 475-481): "Generic credentials, `mailto:`
  raw/decoded state, email diagnostics, and unresolved lexer contexts" (D-C/D-D);
  "FTP-family and scheme-relative posture" (D-B); "Host vocabulary, IDNA

- **S6 evidence** (`evidence/S6-schemes-email.md`): the scheme-by-scheme matrix

## Consequences

- **C-11 becomes jointly closable:** with P2.4 (admission) + P4.1
  (specialization) both accepted, C-11 has a complete typed disposition and G2.A
  can close it.
- **Each admitted scheme has one normative interpretation** keyed on
  `url_standard`, so a v3 draft cannot silently generalize from HTTP(S) and
  mis-specialize ftps, ws/wss, or an opaque scheme (S6 Blocker 1 risk closed).
- **Default unchanged, CRAN-safe:** every specialization is `general`/`whatwg`/
  `rfc3986`-gated; the default `web` + `url_standard = NULL` posture gains no new
  behavior (ADR 0012 D4 byte-identity).
- **The RCON-08 boundary is explicit:** host vocabulary/IDNA/PSL/DNS/IP are
  deferred to the P4 host record, preventing P4.1 from silently pre-empting them.
- **A cleanup backlog is surfaced, not silently absorbed:** five shipped-vs-
  accepted gaps (S6 O3–O7) are named with owners rather than resolved by source
  ordering (S6 High 6 requirement).

## Open questions for the owner

1. **Generic-authority credential preservation + serialization/security (S6
   O3/O9).** P4.1 proposes preserving parsed non-web authority userinfo (fixing
   the dropped-userinfo gap) and defers the clean-vs-exact serialization + secret-
   in-clean-output policy to the P2 serialization decision. Confirm the preserve
   posture, or set a reject/exact-serializer-only alternative here.
2. **`get_password()` selector parity (S6 O4).** Add `scheme_acceptance` /
   `url_standard` selectors to `get_password()` to match `get_user()`/
   `get_userinfo()` (proposed), keep password intentionally web-only, or replace
   the credential-accessor model — routed to an implementation-cleanup slice.
3. **`mailto:` first-recipient raw vs decoded (S6 O5).** P4.1 proposes decode-
   exactly-once with corrected `get_user()` docs. Confirm, or specify raw / dual
   raw+decoded fields; align docs and tests accordingly.
4. **Email accepted-vs-shipped gaps (S6 O6/O7).** Confirm that the `indeterminate`
   unresolved-lexer contract and the URL-level email facts (`userinfo-form`,
   `public-suffix-known`, `smtp-envelope-wire-mode`) are the P4-email slice's to
   resolve against the email PRD, not this record's.
5. **Scheme-relative + `whatwg`-profile precedence (S6 O2).** P4.1 treats
   scheme-relative handling as an independent axis and leaves the
   profile-vs-direct `scheme_policy` divergence to the standards/profile-precedence
   slice (as P2.4 Open Question 5 does). Confirm that split, or fold it in.
6. **FTP-family posture reference vs re-decision.** Confirm P4.1 should merely
   *reference* the resolved ftps-grandfathered / sftp-general-only / no-standards-
   backed-default-port rule (ADR 0012 Open-Q2 / RURL-lvrfrkfa) as done, rather
   than re-decide it.
