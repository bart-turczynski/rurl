# P2.3 — Validation verdicts and legacy status migration

## Relationship to P1.1

This record is the **P2 half of contradiction C-07** (RCON-04). P1.1 (the P1
half) has already proposed the v3 **state/status model** — the layered
*architecture*. P2.3 does **not** redefine that model; it **consumes** it and
adds the two things P1.1 deliberately left to P2: the concrete **verdict
semantics** (enum membership, fatality classification) and the **legacy-status
migration mechanics** (the projection collapse table and the deprecation plan
for `parse_status`). The joint C-07 disposition is therefore `P1.1 + P2.3`
taken together; neither record settles C-07 alone.

P1.1's interface is treated here as **fixed and given** (cited, never altered):

- **(i)** three independently addressable status layers — **L1 syntax/parse
  verdict**, **L2 policy/admission verdict**, and **L3 optional PSL-annotation
  facts**;
- **(ii)** **L3 cannot alter L1/L2**; syntax and policy are computed *before*
  annotation and are invariant under cache warmth, PSL section/list/engine
  choice, and dependency availability;
- **(iii)** optional annotations are **typed results** carrying an explicit
  state (at least the S7-F3 set — `not-requested`, `not-applicable`, `known`,
  `unknown`, `invalid-input`, `dependency-error`) plus a value/provenance
  payload, never a bare `NA`;
- **(iv)** legacy `parse_status` is a **total, pure projection**
  `π(L1, L2, L3)` and is the **sole locus** at which a PSL fact reaches a
  status-shaped output.

Everywhere below, "the model" means P1.1's (i)–(iv). P2.3 fills it in; it does
not fork it.

### 1. Verdict semantics — layer enums and fatality

The eight shipped `parse_status` values (`R/status-constants.R`) are **not a
single flat enum**; they are a *collapsed projection* of three distinct
verdicts. P2.3 assigns each shipped value to its originating layer and gives
each layer its own enum and fatality classification. **Fatality is a property
of L1 and L2 only; L3 is never fatal** (that is (ii): L3 cannot alter L1/L2).

**L1 — syntax / parse verdict.** Enum: `{ pass, fail }`.

- `pass` — the input is structurally parseable under the selected standard
  (curl / general route succeeds; `curl_ok` in
  `.derive_parse_status_vec()`, `R/parse-phases.R`).
- `fail` — **FATAL.** Structural parse failure (`!curl_ok`). No usable object
  is produced.

**L2 — policy / admission verdict.** Enum:
`{ admitted, admitted-scheme-relative, admitted-ftp, rejected-scheme, warn-userinfo }`.
L2 governs *whether an already-parsed object is admitted*, and with what
non-fatal notes, under the active `scheme_acceptance` / `scheme_policy` /
`scheme_relative_handling` / userinfo policy.

- `admitted` — accepted, no note.
- `admitted-scheme-relative` — accepted; the input was scheme-relative and
  `scheme_relative_handling == "keep"` retained it (shipped
  `ok-scheme-relative`). An **accept sub-state**, not a warning.
- `admitted-ftp` — accepted; FTP-family scheme retained under
  `protocol_handling != "strip"` (shipped `ok-ftp`). An **accept sub-state**.
- `rejected-scheme` — **FATAL (admission reject).** Under
  `scheme_acceptance == "web"` an unsupported scheme-bearing token is demoted
  (`looks_like_protocol & !original_has_allowed_scheme & !looks_like_host_port`,
  ADR 0012 D3). This is a **policy** rejection, distinct from an L1 structural
  failure, even though both collapse to shipped `error`.
- `warn-userinfo` — **WARNING (accept-with-note).** Scheme-less,
  userinfo-shaped input (`user@example.com`, `warning-userinfo`): host / domain
  / tld / user resolve, but `rurl` refuses to fabricate `clean_url` (it is
  `NA`) and the row is non-joinable. Accepted with a note; **not** fatal.

**L3 — optional PSL-annotation facts.** Not a fatality axis at all. The three
shipped PSL warnings (`warning-no-tld`, `warning-invalid-tld`,
`warning-public-suffix`) are **registrability facts about the host**, derived
from `tld` / `domain` (themselves PSL outputs). Under (ii) they **must not**
change L1 or L2. They are re-expressed as the **typed annotation state** of the
registrable-domain annotation (§2), and reach a status-shaped output *only*
through π (§4). See §1-summary table.

**Summary — shipped value → layer + fatality (this is the C-07 map):**

| Shipped `parse_status` | Layer | Layer verdict | Fatal? |
|---|---|---|---|
| `error` (from `!curl_ok`) | L1 | `fail` | **FATAL** (structural) |
| `error` (web unsupported-scheme demotion) | L2 | `rejected-scheme` | **FATAL** (admission) |
| `warning-userinfo` | L2 | `warn-userinfo` | no (accept-with-note) |
| `ok-scheme-relative` | L2 | `admitted-scheme-relative` | no (accept) |
| `ok-ftp` | L2 | `admitted-ftp` | no (accept) |
| `ok` | L1 pass + L2 admitted | `pass` / `admitted` | no |
| `warning-no-tld` | L3 | annotation `unknown` (no dot) | **never** (L3) |
| `warning-invalid-tld` | L3 | annotation `unknown` (dot, no suffix) | **never** (L3) |
| `warning-public-suffix` | L3 | annotation `known`, suffix-only | **never** (L3) |

The single most consequential correction C-07 forces: **shipped `error`
conflates an L1 structural failure with an L2 admission rejection**, and the
three PSL `warning-*` values are **L3 facts wrongly shaped as parse verdicts**.
P1.1's model separates them; P2.3 pins which is which.

### 2. Annotation-state resolution (S7-F3) and companion-helper surface

The registrable-domain / PSL annotation returns exactly the S7-F3 typed states
(iii). Resolution rules (transferring S7-F3 substance, `S7-host-idna-psl-ip.md`
§S7-F3 and the "annotation result and validity invariants" clause):

| Annotation state | When it is decided |
|---|---|
| `not-requested` | The PSL annotation was not requested for this call (lazy / companion not invoked). Default main-table path may leave it unrequested; it is never silently `known`. |
| `not-applicable` | The host form has **no** registrable-domain concept: IP-literal host, `file`-scheme host, opaque / general non-special-authority host (ADR 0012 D5). **`not-applicable`, not `unknown`** (S7 IP clause). |
| `known` | A PSL match produced a registrable domain (`tld` and `domain` both non-empty), **or** the input is a known public suffix with no registrable domain above it (suffix-only — shipped `warning-public-suffix`). Suffix-only is a *known fact*, not an error. |
| `unknown` | The host is reg-name-shaped but no public suffix matched: a single-label host with no dot (shipped `warning-no-tld`) or a dotted host whose suffix is unlisted (shipped `warning-invalid-tld`). |
| `invalid-input` | The host is malformed for the annotation's own policy (rejected by the annotation, not by URL syntax). |
| `dependency-error` | `pslr` / the resolved engine is unavailable or errored. Distinct from `unknown`: a lookup that *could not run* is not a lookup that *ran and found nothing*. |

**Invariance (iii + ii).** L1 and L2 are decided *before* the annotation runs
and are **invariant** under annotation selection, cache warmth, PSL
section/list/engine identity, and `pslr` availability. Switching engine,
skipping the lookup, or a `dependency-error` may change **only** the annotation
state and its downstream π projection — never L1/L2.

**Companion-helper surface (ADR 0006 — companion helpers only; MUST NOT widen
`safe_parse_url` columns).** The typed annotation state and the L1/L2 verdicts
are surfaced through the existing companion helpers, not by adding columns to
the parse frame:

- `get_url_diagnostics()` — the aggregator — gains the typed annotation state
  and the L1/L2 verdict fields in its returned diagnostic data frame (it is
  already a separate companion frame, so this does not widen
  `safe_parse_url`).
- `get_host_type()` / `get_scheme_class()` — unchanged in role; they classify
  host shape and scheme class and feed the diagnostic aggregation. Host-form
  classification is what decides `not-applicable` vs an eligible reg-name.
- The parse frame keeps its 18 public fields (`R/utils.R`); the layered
  verdicts live in the companion surface, honoring ADR 0006.

### 3. Resolution / reference behavior (`resolve_url`, RCON-04)

Reference resolution carries the **same three-layer model**, applied to three
independent objects, not one blended `NA`:

- **Base-URL verdict** — L1/L2 of the base. A base that fails L1 makes the
  resolution unresolvable.
- **Reference verdict** — L1/L2 of the relative/absolute reference.
- **Merged-output verdict** — L1/L2 of the resolved absolute URL (the RFC 3986
  recomposition, `R/resolve.R`, then re-parse). Resolution *success* is an **L1
  fact** about the merged output; admission of the merged output is **L2**.

`resolve_url()` currently returns `NA` on unresolvable input or parse failure
and couples its public result to `clean_url` (dropping fragment / credentials,
`R/resolve.R`). The **verdict layering** proposed here is independent of that
output-shape question: whatever the eventual public return contract (an
output-contract decision, RCON-03, not settled here), the resolved object's
L1/L2 verdicts and its host's L3 annotation state follow §§1–2. The current
`NA`-on-failure behavior is an **L1-fatal** projection and is preserved for
back-compat; whether `resolve_url` should expose the merged-output verdicts via
a companion helper is an **open question** (§Open questions Q4).

### 4. Legacy-status migration — the π collapse table

Per (iv), `parse_status` is retained as the **total, pure projection**
`π(L1, L2, L3)` and is the **sole** place a PSL fact reaches a status-shaped
output. π is defined by a strict **error-over-warning, first-match** precedence
ladder. It reproduces `.derive_parse_status_vec()` **exactly** — this table is
the contract that makes the migration byte-identical (§migration).

**π(L1, L2, L3) — evaluate top to bottom; first match wins:**

| # | Condition | Projected `parse_status` |
|---|---|---|
| 1 | L1 = `fail` | `error` |
| 2 | L2 = `rejected-scheme` (admission reject) | `error` |
| 3 | L2 = `warn-userinfo` | `warning-userinfo` |
| 4 | L3 annotation = `unknown`, host has **no** dot | `warning-no-tld` |
| 5 | L3 annotation = `unknown`, host **has** a dot | `warning-invalid-tld` |
| 6 | L3 annotation = `known`, **suffix-only** (no registrable domain) | `warning-public-suffix` |
| 7 | L2 = `admitted-scheme-relative` | `ok-scheme-relative` |
| 8 | L2 = `admitted-ftp` | `ok-ftp` |
| 9 | otherwise (L1 pass, L2 admitted, L3 `known`/`not-applicable`/`not-requested`) | `ok` |

**Precedence notes (the C-07 "which wins" rules):**

- **error-over-warning:** any fatal layer (rows 1–2) collapses to `error`
  before any warning is considered. L1-fatal is checked before L2-fatal, but
  both project to the same `error` token — the *distinction* survives only in
  the L1/L2 companion verdicts, not in `parse_status`. This is the documented
  **loss** in the lossy projection.
- **which PSL warning wins:** by construction (the `host_has_dot` /
  `tld_empty` / `domain_empty` cascade in `.derive_parse_status_vec()`) the
  three PSL states are **mutually exclusive** — at most one applies to any host.
  Precedence among them (rows 4→5→6) is therefore documentation of a
  partition, not a tie-break. Ordering when read as a ladder: `no-tld`
  (no dot) precedes `invalid-tld` (dot, no suffix) precedes `public-suffix`
  (dot, suffix, no registrable domain).
- **userinfo over PSL:** `warn-userinfo` (row 3) outranks any L3 warning,
  matching shipped behavior (a scheme-less userinfo input yields
  `warning-userinfo`, not a PSL warning).
- **`not-applicable` / `not-requested` project to nominal:** an IP / opaque /
  file host (annotation `not-applicable`) and an un-run lookup
  (`not-requested`) both fall through to row 9 `ok` — they are **not**
  warnings. This is exactly why shipped IP / general / file rows are `ok`
  despite empty `tld`/`domain` (the `is_ip_host` / `is_file` / `is_general_ok`
  promotions in `.derive_parse_status_vec()`).
- **`dependency-error` (proposed):** a PSL lookup that could not run projects
  to row 9 `ok` — it must **not** manufacture a `warning-*` (that would let
  dependency availability change the status, the exact S7-F3 hazard). The
  companion annotation state records `dependency-error` honestly. (Owner
  question Q3 confirms this choice.)

### 5. Deprecation and migration plan for `parse_status` / `get_parse_status()`

Honoring the owner mandate that **correctness beats back-compat** — the layered
L1/L2/L3 verdicts are **the truth**; `parse_status` is the **lossy compat
view** — the migration is:

1. **Retain-as-projection (3.0).** The `parse_status` column and
   `get_parse_status()` accessor are **kept** and defined as `π(L1, L2, L3)`
   (§4). No new fatal/warning tokens are added; the vocabulary is frozen at the
   eight shipped values.
2. **Back-compat guarantee (3.0).** For every input, `parse_status` (column)
   and `get_parse_status()` are **byte-identical** to the v2 output, because π
   reproduces `.derive_parse_status_vec()` exactly (§4 is the proof
   obligation, verified by G3 golden fixtures). Existing callers, including
   `canonical_join()`'s `join_parse_status` filter and the
   `.is_ok_status` / `.is_warning_status` / `.is_joinable_status` predicates,
   are unaffected.
3. **New layered accessor (companion, ADR 0006).** A new companion helper —
   **proposed name `get_url_verdicts()`** (naming is Q1) — exposes the L1, L2,
   and typed L3 annotation states as separate columns, so callers that need the
   *un-collapsed* truth (e.g. to tell an L1 structural failure from an L2
   admission rejection, both `error` today) can get it **without** widening
   `safe_parse_url`. `get_url_diagnostics()` aggregates these (§2).
4. **`parse_status` reframed as compat, not deprecated in 3.0.** `parse_status`
   is **documented** as a lossy compatibility projection of the layered
   verdicts (its `man/` entry and NEWS gain that framing), but is **not**
   soft-deprecated in 3.0 — a hard deprecation with a defined removal window is
   deferred to the owner (Q2). The layered accessor is additive; nothing is
   removed in 3.0.
5. **Loss is documented, not hidden.** The one guaranteed information loss —
   L1-fail vs L2-reject both → `error` — is stated in the `parse_status`
   documentation, with a pointer to the layered accessor for callers that must
   distinguish them.

## Rationale

- **C-07 / RCON-04** require that `parse_status`'s conflation of parse success
  with PSL-derived warnings be resolved by *deciding the v3 status model and
  preserving or migrating the legacy projection explicitly*
  (reconciliation §5 C-07; §4 RCON-04, "Validation and recovery need
  independent verdicts and provenance"). §§1,4,5 do exactly that: they name the
  three verdicts, classify fatality, and pin `parse_status` as the explicit,
  documented lossy projection rather than the core verdict.
- **S2-01** (CRITICAL, `S2-standards-validation-repair.md` §S2-01): a
  reconstruction may satisfy loose wording with one overloaded `status` field or
  infer validity from `parse_status`. §1's per-layer enums and the explicit
  "shipped `error` conflates L1 and L2" finding forbid that collapse; §5.5's
  documented loss makes the projection honest.
- **S2-05 / S7-F3** (HIGH / BLOCKER): optional annotation validity has no
  required state model, and a lazy lookup skipping could change what looks like
  the parse verdict (`S2-…` §S2-05; `S7-host-idna-psl-ip.md` §S7-F3 and its
  "annotation result and validity invariants" clause). §2 adopts the exact
  S7-F3 typed states, and §4's `not-requested` / `not-applicable` /
  `dependency-error` → `ok` rules guarantee L1/L2 invariance under annotation
  selection, engine identity, cache warmth, and dependency availability.
- **S2-08 / RCON-04 resolution** (MEDIUM): reference resolution must be
  reconstructed with independent verdicts, not inherited as one blended value
  (`S2-…` §S2-08). §3 gives base / reference / merged-output their own L1/L2
  verdicts and keeps that layering independent of the unsettled output shape.
- **§10 direct verification:** the reconciliation re-verified that
  `parse_status` depends on PSL-derived `tld`/`domain` warnings via
  `.derive_parse_status_vec()` (`R/parse-phases.R`) — the precise mechanism §4's
  π table reproduces byte-for-byte.
- **Owner mandate** (correctness beats back-compat): the layered verdicts are
  the truth and `parse_status` is the lossy compat view — §5 encodes this
  directly, keeping the compat view byte-identical while making the truth
  additively available.

## Consequences

- **G2.A** can mark C-07 disposed once both `P1.1` and `P2.3` are accepted; the
  contradiction register's `owner_decision_ref` already names both.
- **G3** inherits a closed verdict/fatality enum (§1) and the exact π table
  (§4) as contract-matrix rows; the conformance corpus (S2-03) can assert the
  three verdicts independently instead of matching one collapsed token.
- **Implementation (post-acceptance, not this record):** `.derive_parse_status_vec()`
  is refactored to compute L1/L2/L3 first and derive `parse_status` as π, with a
  golden test proving byte-identical output; the layered accessor and the
  `get_url_diagnostics()` fields are added under ADR 0006 (no new parse-frame
  columns).
- **Callers** gain a way to distinguish L1-structural-failure from
  L2-admission-rejection (indistinguishable in `parse_status` today) via the
  layered accessor, without any breaking change.
- **Annotation laziness becomes safe:** because `not-requested` /
  `not-applicable` / `dependency-error` all project to nominal, enabling,
  disabling, failing, or switching the PSL engine can never move `parse_status`
  between `ok` and a warning for reasons unrelated to the host's actual
  registrability.

## Open questions for the owner

- **Q1 — layered accessor name.** Is `get_url_verdicts()` the right name for
  the companion helper exposing L1/L2/L3, or should the layers be folded solely
  into `get_url_diagnostics()` columns with no standalone accessor? (Naming and
  count of new exports.)
- **Q2 — `parse_status` deprecation schedule.** 3.0 retains `parse_status` as a
  documented lossy projection with **no** removal date. Does the owner want a
  soft-deprecation warning and a named removal window (e.g. 4.0), or an
  indefinite compat guarantee? The correctness-over-back-compat mandate permits
  eventual removal; this record does not schedule it.
- **Q3 — `dependency-error` projection.** P2.3 projects a PSL
  `dependency-error` to `ok` (row 9) so dependency availability never changes
  `parse_status`. Confirm this over the alternative (a distinct new
  status token), which would break the frozen eight-value vocabulary and the
  byte-identical guarantee.
- **Q4 — resolver verdict surface.** Should `resolve_url()` expose the
  base / reference / merged-output L1/L2 verdicts via a companion helper, or
  continue to signal only success/failure through `NA`? Tied to the unsettled
  `resolve_url` output-shape decision (RCON-03).
- **Q5 — L1/L2 `error` split visibility.** Confirm that the L1-fail vs
  L2-reject distinction is acceptable to expose **only** in the companion
  verdicts (keeping `parse_status`'s single `error` token), rather than
  introducing a new distinguishing status value.
