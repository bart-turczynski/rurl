# P2.1 — Fixer/recovery ordering and strict/compatibility/repair entry points

## Decision (PROPOSED)

> This record is **proposed, not binding**. It carries `state: PROPOSED`; only the
> owner's merge into `main` moves it to `ACCEPTED` (P0.1 §Decision 3–4). Until then
> it presents the agent's proposed disposition of contradictions **C-02** and
> **C-03** and the entry-point/ordering model behind them for the owner to ratify
> or adjust. The frontmatter reads `PROPOSED` and `accepted_at: pending` because
> the record never claims an approval that does not yet exist.

This record settles reconciliation §8 **P2** item *"Strict, compatibility, and
repair entry points and their ordering"* to the extent needed to dispose the two
contradictions bound to it (C-02, C-03), and defines the ordered pipeline through
which raw input flows. It does **not** define the validation verdicts themselves
(that is the sibling P2-tier validation-verdicts decision, register-bound as
`P2.3`); it defines *where* repair and recovery sit relative to parsing and
validation, and *which entry point* performs each.

### 1. Three entry points (repair posture is an explicit axis)

v3 exposes exactly three parse **entry points / postures**, generalizing the
already-shipped `fixup_posture` seam (ADR 0012 standing rule 2; browser-fixer PRD
Part 1) and the `scheme_policy` acceptance axis (ADR 0010) into one named
repair-posture axis. They are ordered by how much they touch the input:

1. **`strict`** — no repair and no recovery. Raw bytes flow through
   standards-required preprocessing (*parser behavior only*) into a strict parse;
   any input that is not conformant to the selected standard **rejects**. The
   original-input conformance verdict is authoritative (RCON-04; S2-01/S2-02). A
   strict parse never silently validates a rewritten string.

2. **`compatibility`** — the v2-behavior-preserving posture. It performs the
   legacy leniencies rurl 2.x shipped on its default path (default scheme
   inference; the direct-selector repeated-`@` recovery of §3 below) so existing
   callers are not silently broken, but **every such allowance is
   compatibility-only and is not evidence of conformance** (S2-01). It is a
   distinct posture, not a synonym for `strict`.

3. **`repair`** — explicit, opt-in, provenance-logged fix-up. The bounded browser
   fixer (`fixup_posture="browser"` / `profile="browser"`) is the flagship. Repair
   may rewrite bytes; **every repair rule has a stable ID, records before/after,
   and the repaired candidate is re-validated independently of the original**
   (RCON-04; S2-02 "repair off must be a tested no-op"). Repair is never implicit.

These postures are the "strict versus repairing entry points" and "repair
authorization and ordering" that RCON-04 (§4) names as open owner decisions.

**Default posture — RESOLVED at the 2026-07-23 ratification review (B1).** The
v3 **public default is `strict`**: it is the honest, spec-exact contract and the
one that returns an original-input verdict. The rurl 2.x **shipped default** —
scheme inference plus the silent direct-RFC recovery of §3 — is reclassified as
the `compatibility` posture (still selectable, no longer the default) and is
**not** `strict`. This is the 3.0 major-boundary default flip, per the owner's
spec-exactness-over-back-compat mandate; it is coordinated with (and consistent
with) the sibling P2.4 admission-lattice record, which defers the global default
to this decision. (See §Owner ratification.)

### 2. The ordered pipeline (raw input → verdicts)

One deterministic order governs every posture. A posture turns individual stages
on or off; it never reorders them:

1. **Repair (opt-in; `repair` posture only).** If a repair posture is active, the
   bounded fixer runs **first, as one complete pass on the raw string** — outer
   C0/space trim → `;`→`:` for a recognized special scheme → `://` insertion for
   an authority-table scheme (browser-fixer PRD Part 1; `.apply_browser_fixup_vec`,
   `R/parse-phases.R:954-984`, invoked at `:1010`). Provenance is logged. A
   byte-for-byte no-op under `strict`/`compatibility`. **This ordering is the C-02
   disposition (§3).**
2. **Standards-required preprocessing (*parser behavior*, all postures).** WHATWG
   control-character stripping, literal-backslash recognition, and UTS-46 domain
   separator mapping run as **step 1 of the selected standard's parse algorithm**,
   on the (possibly repaired) string (`R/parse-phases.R:1016`, `:1024`, `:1032`).
   These are parser behavior, never fixer policy (browser-fixer PRD "Central
   insight"); they run **after** the fixer, never interleaved into it.
3. **Scheme admission and inference.** `scheme_acceptance` (web/general) gate, then
   `scheme_policy` (`require` under `strict`; `infer` under `compatibility`). The
   fallback-`http` prepend lives here — the single shared inference seam
   (ADR 0012 D4), which is the fixer's "step 4" (browser-fixer PRD Part 1),
   deliberately *not* inside the fixer.
4. **Recovery (`compatibility`/`repair` only; NEVER `strict`).** Selector-scoped
   recoveries — chiefly the repeated-`@` authority recovery
   (`.encode_excess_authority_at_vec`, `R/parse-phases.R:294-323`, called at
   `:1174`) — run here. Under `strict` this stage is a no-op and a non-conformant
   authority rejects. **This binding is the C-03 disposition (§3).**
5. **Strict parse and component extraction.**
6. **Validation verdicts.** Independent verdicts (original-input conformance,
   parse success, scheme admission, repaired-input conformance when repair ran)
   are produced by the P2.3 validation-verdict model, which this record hands off
   to. Both original-input and repaired-input verdicts are exposed (RCON-04).

**Load-bearing invariant.** *"Repair/recovery off" is a tested no-op* (RCON-04;
S2-02): under `strict`, stages 1 and 4 do nothing, the original bytes are
evaluated, and no hidden recovery runs before curl. Every stage that mutates the
string records category, owning standard/policy, before/after span, and whether
it changes conformance — the ordered intervention ledger RCON-04 mandates. This
answers the RCON-03 conflation (source reproduction vs. standards serialization
vs. cleaning vs. display) at the *input* end: repair and recovery are transformations
with provenance, not silent parser internals, and are attributable to a named posture.

### 3. Contradiction dispositions

#### C-02 — browser-fixer phase ordering → **shipped ordering wins; PRD prose superseded**

- **Competing claims.** The accepted browser-fixer PRD's Part 1 bullet list places
  "the selected WHATWG preprocessing" *between* the outer trim and the remaining
  named fixer steps (`design/prd/browser-fixer-and-url-search-classifier.md:41-50`).
  The shipped code runs the **complete** browser fixer first
  (`.apply_browser_fixup_vec` at `R/parse-phases.R:1010`), **then** WHATWG control
  stripping / backslash recognition / separator mapping (`:1016`, `:1024`, `:1032`)
  (S2 finding **S2-09**; reconciliation §5 **C-02**).
- **Chosen ordering (traced).** The **browser fixer runs as one complete,
  contiguous pre-parse pass in front of the parser; standards-required WHATWG
  preprocessing then runs as parser behavior on the fixer's output.** This is the
  shipped ordering above (pipeline stages 1 → 2), verified in code at
  `R/parse-phases.R:954-984` and `:1002-1033`.
- **Why this wins (correctness, not "because it ships").** The accepted layering
  (ADR 0012 standing rule 2; browser-fixer PRD "Central insight") is that `browser`
  is a *fix-up posture layered in front of a spec parser that never guesses.* A
  complete fixer pass in front of the parser embodies that layering; interleaving
  parser-required preprocessing into the *middle* of the fixer braids parser
  behavior into fix-up policy and violates it. The PRD is itself internally
  inconsistent: its own numbered "Ordering" block lists the four fixer steps
  **contiguously** with no interleaved WHATWG step
  (`browser-fixer-and-url-search-classifier.md:74-77`), matching the shipped code;
  only the earlier bullet list interleaves, and it flags that step as "parser
  behavior, not fixer policy — named here only for ordering." The interleaved
  prose is therefore a drafting artifact, not a design choice. Reconciliation §3
  forbids promoting shipped behavior merely because it exists; the justification
  here is the accepted *layering*, which shipped code honors and the interleaved
  prose contradicts.
- **Label for the loser.** The PRD Part 1 **bullet-list** ordering (interleaved
  WHATWG preprocessing) is **superseded** — a documentation defect, not a
  compatibility surface. It is *not* labeled compatibility-only, because the two
  orderings produce the **same final string** for the phase-sensitive inputs (e.g.
  `http;\texample.com`); they differ only in phase *provenance* (S2-09 minimal
  example). There is no divergent output to preserve behind a compatibility flag,
  so the correction is purely to the narrative and the phase-attribution used by
  diagnostics/audit traces. On acceptance, the PRD Part 1 bullet list should be
  corrected (or annotated superseded) so it reads as the contiguous numbered
  "Ordering" block already does.

#### C-03 — repeated-`@` RFC recovery → **classified as REPAIR; bound to `repair`/`compatibility`, excluded from `strict`**

- **Competing claims.** A direct RFC selector recovers repeated raw `@`
  (`http://username@@@@example.com` / `scheme://username@@@@example.com`) and
  reports `ok` (`tests/testthat/test-url-standard-authority.R:14-23`), while
  RFC-general rejects the same grammar
  (`tests/testthat/test-general-acceptance.R:51-66`) (S2 finding **S2-02**;
  reconciliation §5 **C-03**; corroborated by S3 line 130). The recovery is
  `.encode_excess_authority_at_vec` (`R/parse-phases.R:294-323`, called `:1174`),
  which treats the **last** `@` as the userinfo/host delimiter and percent-encodes
  the earlier `@` bytes.
- **Classification: REPAIR.** RFC 3986 `userinfo` is
  `*( unreserved / pct-encoded / sub-delims / ":" )` and admits no unescaped `@`;
  a raw interior `@` must be percent-encoded (`%40`). The recovery therefore does
  not *recognize* conformant syntax — it **mutates the input bytes** (`@` → `%40`)
  to force a parseable authority. That is byte-changing fix-up with provenance:
  **repair**, not syntax recognition, and not a mere leniency.
- **Bound entry points.**
  - **`strict`**: does **not** perform it. Repeated raw `@` is non-conformant and
    **rejects** — i.e. today's RFC-*general* behavior becomes the correct strict
    baseline. This resolves the direct-vs-general inconsistency in favor of
    strict-rejects, per the owner's spec-exactness-over-back-compat mandate and
    S2-01/S2-02 (a "strict" path must not silently validate a repaired string).
  - **`repair`**: **may** perform it as a named, provenance-logged repair rule,
    with the repaired candidate re-validated independently of the original.
  - **`compatibility`**: **may** retain today's direct-selector recovery behind the
    explicit compatibility posture, **labeled compatibility-only** — preserved for
    v2 direct-RFC callers, but never asserted as RFC conformance.
- **What changes.** The defect being disposed is that the *direct* RFC selector
  silently repairs while *general* rejects: the "strict" path is currently a
  repairing path. Under this decision the silent recovery leaves `strict`
  entirely; it survives only where it is honestly named (`repair`) or explicitly
  bracketed (`compatibility`). Strict RFC and general RFC then agree: both reject
  repeated raw `@`.

## Rationale

- **RCON-04 (§4, lines 154–176)** requires independent verdicts and an *ordered,
  typed intervention ledger* separating "standards-mandated preprocessing,
  normalization, dependency compatibility shims, product compatibility recovery,
  browser repair, cleaning, and display transformation," and names "strict versus
  repairing entry points" and "repair authorization and ordering" as open owner
  decisions. This record supplies the entry-point axis (§1), the fixed stage order
  (§2), and the two contradiction dispositions (§3) that the ledger and verdicts
  build on.
- **RCON-03 (§4, lines 125–152)** names the conflation of source reproduction,
  standards serialization, cleaning, and display. At the input end the same
  conflation appears as "is this operation parsing, or repair?" — §2's invariant
  (repair/recovery are provenance-bearing transformations attributable to a named
  posture, never silent parser internals) is the input-side answer; the output-side
  contracts remain with the sibling P2.2 decision.
- **Reconciliation §5 C-02/C-03 required dispositions** are met verbatim: C-02
  "choose and trace one ordering; label the other superseded or compatibility-only"
  → shipped ordering traced, PRD bullet list superseded (§3 C-02); C-03 "classify
  recovery as syntax, compatibility, or repair and bind it to an entry point" →
  classified repair, bound to `repair`/`compatibility`, excluded from `strict`
  (§3 C-03).
- **Owner mandate (spec-exactness over back-compat; MEMORY: rurl
  spec-exactness-over-backcompat).** Where shipped leniency and standard conformance
  disagree, the strict contract follows the standard; the leniency survives only
  behind an explicit compatibility/repair posture. Both dispositions apply this
  rule.
- **Accepted authority relied on.** ADR 0012 (general parser; `browser` is a fix-up
  posture — standing rule 2 — and D4 one-prepend seam), ADR 0010 (`scheme_policy`
  acceptance axis), and the browser-fixer PRD Part 1 (bounded fixer behavior and
  its own numbered ordering block).

## Consequences

- The v3 spec must expose a **repair-posture axis** with `strict` /
  `compatibility` / `repair` and hold the six-stage pipeline order (§2) invariant
  across all three; a posture may only enable/disable stages, never reorder them.
- **`strict` becomes non-repairing and reject-on-non-conformance**, with an
  original-input verdict; the repeated-`@` recovery and default scheme inference
  are reclassified out of it. This is a behavior change for any future caller that
  selects `strict`, and — if the public default flips to `strict` (open question) —
  a 3.0 default-behavior change for direct RFC callers relying on silent
  repeated-`@` recovery. It is confined to the 3.0 major boundary.
- Phase-provenance / audit traces become stable: the fixer is always attributed as
  a complete pre-parse pass; WHATWG preprocessing is always attributed as parser
  behavior (§3 C-02). Diagnostics keyed on phase no longer depend on an ambiguous
  prose-vs-code narrative.
- Downstream G3 (contract-matrix closure) inherits a defined input-side contract:
  the intervention ledger's "category" column (RCON-04) has concrete values for the
  fixer steps and the repeated-`@` recovery, and "repair off = no-op" is a testable
  invariant for G4.
- The register single-writer (owner) can close C-02 and C-03 at G2.A citing this
  record.
