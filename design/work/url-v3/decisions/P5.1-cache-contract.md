# P5.1 — Authoritative cache contract

## Decision (PROPOSED)

> This record is **proposed, not binding**. It is authored by the agent at
> `state: PROPOSED` under the P0.1 agent-proposes / owner-accepts mechanism. It
> becomes authoritative only when the owner (`github:bart-turczynski`) merges it
> into `main`; until then nothing here is a settled product decision. Open

The rurl 3.0 cache contract has two parts: the **cache model** (which caches
exist, their defaults, their eviction, and the transparency invariant that binds
all of them) and the **disposition of contradiction C-08** (the 100,000-vs-
unbounded documentation drift), together with the documentation-consistency gate
that C-08's required disposition mandates.

### 1. The authoritative cache contract

**1.1 The caches that exist.** rurl memoizes only its *own* work; the Public
Suffix List query cache lives in `pslr`, not here (ADR 0001; `R/zzz.R:1-9`). The
v3 cache surface is exactly the three caches registered in `.CACHE_REGISTRY`
(`R/zzz.R:19-38`), introspected and configured through the three exported
functions `rurl_cache_info()`, `rurl_cache_config()`, and `rurl_clear_caches()`:

| Cache | Memoizes | Default bound | Eviction | Configurable |
|---|---|---|---|---|
| `full_parse` | the option-independent Stage-A parse core (still named `full_parse` for API stability) keyed by url + the Stage-A-affecting axes | **bounded: 100,000 entries** | hard **reset-watermark** — when a new key would exceed the bound, the entire cache is cleared, then the new key is stored | `rurl_cache_config(full_parse=, max_full_parse=)` |
| `puny_encode` | IDNA/Punycode host **encode** round-trips | **unbounded by design** | none | `rurl_cache_config(puny_encode=)` |
| `puny_decode` | Punycode host **decode** round-trips | **unbounded by design** | none | `rurl_cache_config(puny_decode=)` |

The `full_parse` key is the URL plus every Stage-A-affecting axis
(protocol/scheme handling, `www_handling`, `tld_source`, relative, standard,
policy, acceptance, fix-up, and the `pslr` **engine snapshot identity**), so a
change on any of those axes is a distinct entry and can never produce a
cross-profile stale hit (`R/parse.R` `.parse_cache_keys()` /
`.engine_cache_token()`; verified against S8 coverage-map "Full-parse key" and
"Engine identity" rows). The two Punycode caches are keyed by host/label and are
**"small by design"** — bounded by the number of *unique hosts/labels* seen, not
by the URL×option cross-product — which is why they carry no count bound
(`R/zzz.R:187-192`, `:311-316`).

**1.2 Defaults.** The `full_parse` cache is **bounded at 100,000 entries by
default** (`R/zzz.R:81`, `assign("full_parse_max", 100000L, ...)` in `.onLoad`);
`puny_encode` and `puny_decode` are **unbounded by default**; all three caches
are **enabled by default**. `rurl_cache_config(max_full_parse = Inf)` restores
the historical unbounded `full_parse` behavior for callers who explicitly opt in.

**1.3 Eviction is a hard reset-watermark, not LRU/FIFO.** When `full_parse`
reaches `max_full_parse` and a genuinely new key arrives, the **entire** cache is
emptied and rebuilt from scratch (`R/zzz.R` `.cache_set_derived()` lines
205-212). `max_full_parse` therefore caps *peak memory*; it is **not** a
working-set size. This behavior is retained in v3 as the accepted default policy
(an LRU/FIFO alternative is an open question below, not a proposal here).

**1.4 The semantic-transparency invariant (the load-bearing rule).** A cache is
a pure performance/memory optimization and **MUST NOT change any observable
result.** For every rurl operation and every input vector, the output must be
**byte-identical** across all of the following cache states:

- **cold vs warm** — first call vs a repeated call whose keys are already stored;
- **enabled vs disabled** — the same call with the cache toggled off via
  `rurl_cache_config(<cache> = FALSE)`;
- **bounded vs unbounded** — `max_full_parse = 100000` vs `Inf`;
- **pre- vs post-eviction** — a call that straddles a reset-watermark clear vs
  one that does not;
- **scalar-loop vs one vector call vs deterministic chunk recombination** — the
  cache must not make a chunked/vectorized run diverge from a scalar loop.

Only elapsed time and peak memory may differ; **component values, `parse_status`,
warnings, names, order, and types may not.** A `pslr` engine or default-snapshot
change must never yield a stale hit (§1.1). This is the invariant that P5.3's
`self-metamorphic` cache-transparency oracle checks; P5.1 owns the **contract**,
P5.3 owns the **claim/oracle policy** over it.

### 2. Disposition of contradiction C-08

**C-08 (reconciliation §5):** runtime initializes `full_parse` with a
**100,000-entry** bound, while `README.Rmd` says it is **unbounded by default**.
Required disposition: *"Correct one side and add a documentation-consistency
gate."*

**2.1 Which side is authoritative — the code.** The **runtime default of 100,000
entries is the truth**; the `README.Rmd` claim of an unbounded default is
**wrong and must be corrected**. Rationale:

- The bound is a **memory-safety default**: parsing millions of unique URLs must
  not grow the cache without limit (`R/zzz.R:77-81`). An *unbounded* default is a
  latent **OOM / DoS** risk on exactly the high-volume, all-unique-input workload
  the package treats as primary (S8 coverage map, "Full-parse capacity"). The
  safe default is the one shipped.
- The runtime and the **generated help are already internally consistent**:
  `man/rurl_cache_config.Rd` (roxygen at `R/zzz.R:296-317`) documents the
  100,000 default and the reset-watermark correctly. Only the **README prose**
  (`README.Rmd:265-269`) is stale. Correcting the single stale surface to match
  code + help is the minimal, honest repair.
- The owner **correctness-over-back-compat mandate** applies: where a document
  and the shipped safe behavior disagree, the document yields; we do not weaken
  the shipped memory-safety default to preserve a stale sentence.

**2.2 The concrete correction (deferred to implementation, not performed here).**
`README.Rmd:265-269` must be rewritten so that "The `full_parse` cache is
unbounded by default (`max_full_parse = Inf`)" becomes a statement that
`full_parse` is **bounded at 100,000 entries by default** (a hard
reset-watermark that caps peak memory), and that `max_full_parse = Inf` is the
*opt-in* to the historical unbounded behavior. The generated `README.md` is
re-knit from `README.Rmd`. This record only *decides* the correction; the edit is
a G4-tier implementation task governed by the gate in §2.3 and is out of P5.1's
one-file scope.

**2.3 The documentation-consistency gate (becomes a G4 executable check).** Add a
check that **fails whenever the documented bound and the runtime default
diverge**. The gate:

1. reads the **runtime default** — the literal used to initialize
   `full_parse_max` in `.onLoad` (`R/zzz.R:81`), verified equal to the value
   `rurl_cache_config()` reports for `max_full_parse` on a freshly loaded package
   with no prior configuration;
2. reads every **documented bound** — at minimum the `full_parse` default stated
   in `README.Rmd`, in the `rurl_cache_config`/`rurl_cache_info` roxygen (hence
   `man/*.Rd`), the Caching vignette/section if it names a number, **and any
   inline `R/**` source comment that states a cache-bound posture** (e.g. the
   stale `R/zzz.R:55-56` comment calling the caches "unbounded", contradicted by
   the operative `100000L` at `R/zzz.R:77-81` — a second stale surface a
   docs-only scan would miss; drift audit RURL-xgtjmxep);
3. **fails** the gate if the runtime default and any documented bound are not the
   same number (Inf compared as Inf).

The gate is deterministic, needs no network, and runs in the `verify` chain so a
future default change or doc drift cannot re-open C-08 silently. It closes
S8 finding 13 ("current-state documentation contradictions need an
acceptance-time closure rule") for the cache surface specifically. A companion
concern surfaced by S8 High-5 — the cache-policy test helper forces
`max_full_parse = Inf` (`tests/testthat/test-cache-policy.R:7-30`), so the
**shipped 100,000 default is not itself under test** — is recorded as an open
question (§Open questions Q3), since it is a test-coverage decision rather than a
documentation-consistency fact.

## Rationale

- **Reconciliation §5 C-08** states the contradiction verbatim and fixes the
  required disposition ("Correct one side and add a documentation-consistency
  gate"). §2 of this record executes exactly that: pick the code side, correct
  the README, add the gate.
- **Reconciliation §4 RCON-09** requires "a cache contract covering keys, bounds,
  eviction, clearing, external-data versions, and cached/uncached equivalence,"
  and states plainly that cache behavior "is observable but not specified as
  semantically transparent, bounded, deterministic, or versioned by external
  data." §1 supplies each of those cells: keys (§1.1), bounds/eviction (§1.1-1.3),
  clearing (`rurl_clear_caches`, §1.1), external-data version (the `pslr` engine
  snapshot in the key, §1.1), and cached/uncached equivalence (§1.4).
- **Reconciliation §8 P5** names "Cache defaults and semantic-transparency rules"
  as the cell P5.1 fills, and separately lists "Curl-removal sequence and closure
  gate" (kept out of scope per §3).
- **Reconciliation §10** records the direct verification: "Runtime and cache
  documentation disagree … runtime bound 100,000; README says unbounded default"
  against `.onLoad()`/`rurl_cache_config()` in `R/zzz.R` and the caching section
  of `README.Rmd`. §2.1 resolves it in favor of the runtime.
- **Frozen evidence S8** (PRIMARY): the coverage-map "Cache semantics" row rates

## Consequences

- The v3 cache surface is now fully specified: three caches, their defaults, a
  named eviction policy, and a single transparency invariant that P5.3's oracle
  can check and G4 can gate. RCON-09's "cache contract" cell is filled for the
  bounds/eviction/clearing/equivalence dimensions.
- C-08 is closed in favor of the shipped safe default; the memory-safety bound
  stands, the stale README sentence is scheduled for correction, and a G4 gate
  prevents recurrence.
- A caller relying on the *documented* unbounded default gets a **documentation
  correction, not a behavior change** — the runtime already bounded the cache, so
  no shipped behavior changes for anyone; only the description becomes accurate.
- The transparency invariant constrains every future cache change: any
  optimization that would make warm/cold/disabled/bounded output diverge is, by
  this contract, a bug — not a permitted trade-off.
- P5.2 (determinism) and P5.3 (cache-transparency oracle) build on this contract
  without renegotiating it; the curl-removal cell (P5, separate) may reference
  the `full_parse`/percent-cache lineage but is not sequenced here.

## Open questions for the owner

1. **Eviction policy.** §1.3 retains the shipped **hard reset-watermark** (empty
   the whole cache on overflow) as the v3 default. Is that acceptable, or should
   v3 move `full_parse` to an **LRU/FIFO working-set** eviction (steadier hit
   rate on long unique streams, at the cost of per-entry bookkeeping)? P5.1
   proposes *keep the reset-watermark* and treat LRU as a deferred enhancement.
2. **Bound units.** The bound is an **entry count** (100,000), not a **memory
   ceiling**. On very large parsed objects the peak *bytes* vary with entry size.
   Should v3 keep count-based bounds (simple, deterministic) or add an optional
   memory-based ceiling? P5.1 proposes *keep count-based*; a memory ceiling is a
   separate performance-budget decision (P5 performance cell).
3. **Testing the shipped default.** The cache-policy test helper forces
   `max_full_parse = Inf` (`tests/testthat/test-cache-policy.R:7-30`), so the
   **100,000 default and its reset-watermark are not tested as defaults**. Should
   the G4 gate (or a new test) also assert the shipped default and one
   reset-watermark cycle? P5.1 proposes *yes — add default-and-reset coverage*
   alongside the §2.3 documentation gate.
4. **Punycode cache posture at scale.** `puny_encode`/`puny_decode` are unbounded
   "small by design," but a massive **all-unique-IDN** stream could grow them
   without limit (S8 coverage map, "Punycode caches"). Keep them unbounded (with
   the disable switch as the only relief), or give them an optional bound too?
   P5.1 proposes *keep unbounded + disable switch*; an optional bound is a
   deferred enhancement.
5. **Worker/process cache scope.** Caches are process-local and the `pslr`
   matcher is a non-serializable external pointer (S8 Medium 9;
   `ARCHITECTURE.md:106-128`). The transparency invariant (§1.4) is stated
   per-process. Confirm that parallel/chunked cache *scope* is deferred to the
   P5 parallel/chunk cell and not re-decided here.
