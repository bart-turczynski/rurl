# P0.1 — Owner identity and durable approval mechanism

## Decision

The rurl 3.0 protocol-hardening effort adopts a **tracked, approved-by-merge
decision record** as its durable authority mechanism, established now — before
the general Section 6 record schema (G0.3) exists — so that P0 does not depend
on G0.3 and re-create the RCON-01 durability blocker.

1. **Record form.** Each owner decision is one Markdown file with a YAML
   frontmatter header, stored under the tracked path
   `design/work/url-v3/decisions/`. The frontmatter carries the fields shown in
   this record's own header; the body carries the exact decision text,
   rationale, and consequences. This mirrors the repository's existing ADR and
   design-doc conventions and is importable by G0.3 without rewriting substance.

2. **Identity and authority.** The sole owner and approver is
   **Bart Turczynski `<bartek@turczynski.pl>`**, acting as repository owner. The
   same person holds the "designated approver" (process/evidence records) and
   "designated verifier" roles named in the Section 6 lifecycle until either is
   explicitly delegated by a superseding P0-tier record.

3. **Approval = merge.** A record becomes `ACCEPTED` when, and only when, the
   owner merges it into `main`. The merge commit that first contains the file is
   the durable approval evidence (recoverable with `git log --follow`). A record
   that asserts `state: ACCEPTED` while still on a feature branch is a *proposal
   of the accepted content*; the owner's merge is the ratifying act, exactly as
   an "Accepted" ADR is ratified by merge.

4. **Agent-proposes / owner-accepts.** The agent may author records at
   `DISCOVERED` or `PROPOSED` and open the pull request. Only the owner's merge
   moves a record to `ACCEPTED`. No agent action confers authority.

5. **Tracker-only decisions carry no authority.** Fiberplane (`fp`) issue
   comments, statuses, titles, and parent links are pointers and working notes
   only. **Nothing stated solely in the tracker is normative.** A decision is
   authoritative only once it exists as an accepted record under this mechanism.
   Where a tracker comment and an accepted record disagree, the record governs.

6. **Immutability and supersession.** Accepted records are append-only. Their
   substance is never edited in place; a changed premise is handled by writing a
   new record that names the one it supersedes (via `supersedes` /
   `superseded_by`) and by marking dependents `INVALIDATED` per the Section 6
   lifecycle. History is never silently rewound.

## Consequences

- P0.2 and P0.3 are ratified *under this mechanism* and reference it via
  `depends_on: [P0.1]`.
- G0.3 must import this record and validate it against the general schema
  **without altering its substance**; any schema that cannot represent these
  fields is incomplete, not a licence to rewrite the decision.
- The mechanism is deliberately minimal (no signing infrastructure) because the
  owner is sole approver and the merge commit already binds identity through
  the forge. A future multi-approver need is a new P0-tier record, not an edit.
