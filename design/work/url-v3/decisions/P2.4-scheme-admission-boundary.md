This record settles the **admission** portion of C-11: *where* ADR 0004's closed
web scheme set still governs and *where* ADR 0012's open-scheme model governs. It
defines the v3 **scheme-admission lattice** â€” the predicate that decides which
scheme tokens may enter parsing under which axis settings â€” and the interface the
sibling record **P4.1** (scheme specialization) must honor. It deliberately does
**not** decide what the parser *does* with any admitted scheme; that is P4.1.

### D-A. C-11 admission disposition: `SUPERSEDED`, with a compatibility-only residue

ADR 0004's standing rule â€” *"keep the allowed-scheme set closed
(`http`/`https`/`ftp`/`ftps`); do not add `ws`/`wss`/`file`"* (0004 Â§Consequences)
â€” is **`SUPERSEDED`** as a hard cap on the parser by ADR 0012, which states its
own supersession explicitly (0012 header: *"Supersedes the 'keep the scheme set
closed' standing rule of ADR 0004"*; 0012 Standing-rule 1; 0012 D3). The old
rule is **retained only as a default/compatibility layer**, precisely as C-11's
required disposition directs (*"Apply ADR 0012's explicit supersession precisely
and retain the old rule only where compatibility/default policy still calls for
it"*).

The residue is named exactly:

1. **Where ADR 0004 still governs â€” the `web` acceptance value only.** The closed
   set survives as the concrete membership of the **default `scheme_acceptance =
   "web"` allowlist**, `.SUPPORTED_SCHEMES` (verified `R/utils.R:27` =
   `c("http", "https", "ftp", "ftps", "file")`). Under `web`, a scheme-bearing
   input outside that allowlist is an `error` â€” this is ADR 0004's closed-set
   behavior, now decoupled from `protocol_handling` (ADR 0010) and kept for the
   hard CRAN byte-identity constraint (0012 D4).

2. **Where ADR 0004 no longer governs â€” the parser and the `general` value.**
   Under `scheme_acceptance = "general"` (ADR 0012 D3) the parser admits **any
   syntactically valid scheme token** (`^[A-Za-z][A-Za-z0-9+.-]*$`). ADR 0004 is
   **not** a cap here and must never be reintroduced as one.

3. **ADR 0004's *literal* closed list is itself already narrower than the
   surviving residue.** ADR 0004 said `http/https/ftp/ftps` and *"do not add
   `file`"*; `file` shipped anyway (PR #134/#135, noted in 0012 Context) and
   `ftps` is grandfathered with no standards-backed default port (0012 Open-Q2 /
   RURL-lvrfrkfa, *resolved*). So even the *default* allowlist has moved past
   ADR 0004's wording. The compatibility residue is therefore **the current
   `.SUPPORTED_SCHEMES` membership**, not ADR 0004's original four-scheme list.

**Disposition verdict for C-11 (admission half): `SUPERSEDED` (by ADR 0012) with
a `COMPATIBILITY-ONLY` residue = the default `web` allowlist.** Adding any scheme
to that default allowlist (e.g. promoting `ws`/`wss`) remains a deliberately
versioned widening requiring a new P-tier decision (0012 D4), never a silent
change and never a P4.1 specialization.

### D-B. The v3 scheme-admission lattice

Admission is decided by **three orthogonal axes**, exactly as ADR 0012 D3
enumerates them (extending ADR 0010's three-axis model). Admission is a *distinct
question* from interpretation; "orthogonal" means separate questions and
arguments, **not** that every Cartesian cell is valid (0012 D3).

| Axis | Question it answers | Values | Source |
|---|---|---|---|
| `scheme_acceptance` | Which scheme *tokens* may enter parsing? | `web` \| `general` | ADR 0012 D3 |
| `scheme_policy` | Is scheme-*less* host-shaped input accepted? | `infer` \| `require` | ADR 0010 |
| `url_standard` | Which standard *interprets* an admitted token? | `NULL` \| `rfc3986` \| `whatwg` | ADR 0007 |

**Two named admission classes** (nameable so P4.1 and later records can cite
them):

- **`ADMIT-WEB` (default).** `scheme_acceptance = "web"`. A scheme-*bearing* token
  is admitted **iff** its lowercased form âˆˆ `.SUPPORTED_SCHEMES`
  (`http`/`https`/`ftp`/`ftps`/`file`). Any other scheme-bearing token â†’ `error`.
  This is ADR 0004's surviving residue (D-A). Composes freely with
  `url_standard âˆˆ {NULL, rfc3986, whatwg}`.
- **`ADMIT-GENERAL`.** `scheme_acceptance = "general"`. A scheme-bearing token is
  admitted **iff** it matches `^[A-Za-z][A-Za-z0-9+.-]*$`. This class is ADR
  0012's open-scheme model. **Composition rule (ADR 0012 D3):** `general`
  requires a **non-NULL** `url_standard`; `general + NULL` is an
  option-validation error before any row parses (verified `R/parse.R:1344-1346`:
  `if (opts$scheme_acceptance == "general" && is.null(opts$url_standard)) stop(...)`).

Scheme-**less** input is governed by a *separate* axis (`scheme_policy`) and is
**not** part of the scheme-token admission set â€” C-11 concerns scheme-*bearing*
tokens. `scheme_policy = "infer"` fabricates `http://` for host-shaped scheme-less
input (ADR 0010); `require` rejects it. Scheme-*relative* `//host` is a third,
independent axis (`scheme_relative_handling`, ADR 0010 boundary) and is out of
scope here.

**Admission â‰  parse success.** `ADMIT-GENERAL` admits the *token*; the remainder
may still fail (WHATWG host-missing for a special scheme, D1 RFC generic-grammar
violation, etc.). Those per-scheme failure boundaries are P4.1's, not P2.4's
(0012 D3 fatal-vs-reportable boundary).

#### Admission by input class Ã— axis setting

The cells state **admitted / `error`** at the *admission* gate only; how an
admitted token is *interpreted/specialized* (special vs non-special, host model,
path kind) is P4.1 territory and shown only as an interpretation hint.

| Input class | `web` (default) | `general` + `whatwg` | `general` + `rfc3986` |
|---|---|---|---|
| `http`, `https` | **admitted** (allowlist) | admitted (special hint) | admitted |
| `ftp` | **admitted** (allowlist) | admitted (special hint) | admitted |
| `ftps` | **admitted** (grandfathered; no std default port) | admitted (non-special hint) | admitted |
| `file` | **admitted** (allowlist) | admitted (special hint) | admitted (RFC 8089 overlay â€” P4.1) |
| `ws`, `wss` | **`error`** (not in allowlist; special-ness metadata is admission-inert here) | admitted (special hint; ports 80/443) | admitted |
| `mailto`, `tel`, `data`, `sftp`, arbitrary `foo:` | **`error`** | admitted (non-special hint) | admitted |
| scheme-less host-shaped (`example.com`) | governed by `scheme_policy` (inferâ†’prepend `http://`; requireâ†’`error`) | governed by `scheme_policy` | governed by `scheme_policy` |
| scheme-relative (`//host`) | governed by `scheme_relative_handling` (out of scope) | same | same |

Verified shipped anchors: `web` allowlist `.SUPPORTED_SCHEMES` (`R/utils.R:27`);
`ws`/`wss` special-ness metadata present but admission-inert under `web`
(`.WHATWG_SPECIAL_SCHEMES` `R/utils.R:38`, comment `R/utils.R:44` "no ws/wss
input reaches this recovery path under the default"); composition rule
(`R/parse.R:1344`); `general` route entry (`R/parse.R:1618`
`.general_parse_vec(urls, opts$url_standard, opts$scheme_acceptance)`).

#### Profile / direct-call lattice

`scheme_acceptance` is set two ways (ADR 0012 D6; verified `.URL_PROFILES`,
`R/parse.R:943-989`):

- **Direct call.** `scheme_acceptance`, `scheme_policy`, `url_standard` are set
  independently; defaults are `web` / `infer` / `NULL`. Explicit args win over
  profile expansion (iron rule).
- **Named profiles (ADR 0012 D6).** `browser`, `whatwg`, `rfc-syntax` each set
  `scheme_acceptance = "general"` (verified `R/parse.R:948,958,969`); `seo`
  (alias `canonical`) sets `scheme_acceptance = "web"` (verified
  `R/parse.R:981`). So general admission **is reachable through the shipped public
  `profile=` surface** and through `get_mailto_recipients()`'s default
  `scheme_acceptance = "general"` (public-surface-inventory row 74). The
  in-source comment calling `general` "internal-only â€¦ NOT on any public
  signature yet" (`R/parse.R:854-856,864`) is **stale relative to the profile
  layer that shipped it** â€” see Open Question 4.

The `whatwg`-profile foot-gun (profile sets `scheme_policy = "require"` while a
direct `url_standard = "whatwg"` inherits `infer`, so they diverge on scheme-less
acceptance; 0012 D6) is a **direct-vs-profile precedence** question (S6 O2,
RCON-05 open decision) owned by the standards/profile-precedence slice, **not**
by P2.4. P2.4 fixes only *which axis owns admission*; it does not resolve the
foot-gun policy.

### D-C. v3 default admission

The v3 default admission **stays `ADMIT-WEB`** â€” `scheme_acceptance = "web"`,
`scheme_policy = "infer"`, `url_standard = NULL` â€” i.e. the opt-in model is
**preserved**, byte-for-byte, for the hard CRAN backward-compatibility constraint
(0012 D4). General admission is opt-in via a profile or the explicit
`general` + non-NULL `url_standard` pairing (0012 D4). `ws`/`wss` do **not** join
the default allowlist (0012 D4).

**Coordination (not overridden here).** Whether v3 flips its *global* default
posture is the open question **O1 (v3 default admission)** in the RCON-05 queue,
and it is coupled to the **P2.1 default-posture / strict-vs-repair entry-point**
decision (P2 Â§8: "Strict, compatibility, and repair entry points"). P2.4
**proposes** preserving the `web` default and **defers** any default-flip
authority to that coordinated decision: if P2.1/O1 sets a different global default
entry posture, this record's default-admission value follows it, and the two must
be reconciled at ratification. P2.4 does not pre-empt that choice (Open Question
1).

### D-D. Interface P4.1 must honor (scope boundary)

P4.1 (scheme specialization) defines what the parser *does* with each admitted
scheme; P2.4 defines *which* schemes are admitted and *under which* axis setting.
P4.1 **must honor** this admission interface and may not re-open it:

1. **Admission predicate is fixed and owned here.** A scheme-bearing token is
   admitted **iff** `(web âˆ§ lower(token) âˆˆ .SUPPORTED_SCHEMES) âˆ¨ (general âˆ§ token
   matches ^[A-Za-z][A-Za-z0-9+.-]*$ âˆ§ url_standard â‰  NULL)`. P4.1 may **not**
   re-admit a token `web` rejects, nor reject a token `general` admits, at the
   specialization layer.
2. **Admission â‰  success.** An admitted token may still fail on its remainder;
   P4.1 owns those per-scheme fatal boundaries (ftp-family posture, ws/wss
   inertness-vs-special, `file:` RFC-8089 overlay, mailto raw/decoded, generic
   credentials, unresolved lexer contexts) â€” but only *within* the admitted set.
3. **Admission chooses neither interpretation nor specialization.** special vs
   non-special, host model, path kind, default ports, transform eligibility are
   **not** admission facts; they are interpretation (`url_standard`) + P4.1.
4. **The `web` allowlist is a closed, compatibility-frozen set.** P4.1 must not
   treat any non-`web` scheme as admissible under `web`. Widening the default
   allowlist is a versioned P-tier decision, never a P4.1 specialization.
5. **ws/wss metadata is admission-inert under `web`.** `.WHATWG_SPECIAL_SCHEMES`
   knows ws/wss special-ness and default ports, but under `web` no ws/wss row
   survives admission; P4.1 specializes ws/wss **only** under `general`.
6. **The `mailto:` PSL carve-out is a specialization, not an admission widening.**
   `mailto:` is admitted only under `general`; its recipient-domain PSL carve-out
   (0012 D7) is P4.1's to define and does not alter the admission predicate.

## Consequences

- **C-11's admission half becomes decidable and precise:** ADR 0004 is a
  compatibility residue (the `web` allowlist), not a parser cap. Reviewers can
  point to one predicate (D-B) for "is this scheme admitted?".
- **Default unchanged, CRAN-safe:** D-C preserves the `web`/`infer`/`NULL`
  default byte-for-byte; no default-posture caller sees a new `mailto:`/`ws:` row.
- **P4.1 is unblocked with a crisp, nameable interface** (`ADMIT-WEB` /
  `ADMIT-GENERAL`, the admission predicate, admission â‰  success), so
  specialization work cannot silently re-litigate admission.
- **A versioned-widening tripwire is established:** promoting any scheme into the
  default allowlist is now explicitly a new P-tier decision, closing the "silent
  widening" risk S6 Blocker 1 warns about.
- **New maintenance surface:** the admission lattice must stay in sync with
  `.SUPPORTED_SCHEMES`, the composition-rule guard, and the profile bundles; G3's
  contract matrices are the mechanical check.
