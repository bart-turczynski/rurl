This record settles reconciliation §8 **P3** items *"Key equivalence,
representation, collisions, invalid rows, and versioning"* and *"Six-join
semantics and `canonical_join()` migration"*, and disposes contradiction
**C-06**. It defines the independent v3 comparison key, its versioned policy and

**not** define the clean-presentation surface (`get_clean_url` / `clean_url`)
itself — the sibling output-contracts proposal **P2.2** establishes that

it; P2.2 is cited here as a sibling proposal but is deliberately **not** a
`depends_on` edge (it is not on this branch).

### D-A. The independent v3 comparison key contract

The v3 comparison key is a **first-class, non-URL projection derived from the
canonical identity state — the internal structural/identity record produced
*after* standard interpretation but *before* any cleaning or display
transform** (S5 HIGH 3 concrete correction, lines 174-190; `evidence/S5-keys-joins.md`).
It is **never** derived from `clean_url` or any presentation string. This is the
load-bearing inversion C-06 demands: identity precedes and is independent of
presentation.

1. **Public projection and vector contract.** A new exported
   `get_url_key(url, policy = url_key_policy())` (S5 API row, line 304). It is
   length-preserving and names-preserving; character/factor input is accepted
   under explicit coercion rules; the `policy` object is not silently recycled.
   `NAMESPACE` currently exports no such symbol (S5 BLOCKER 1, line 140) — this
   contract creates it.

2. **Key representation (classed, versioned, injectively framed).** The key is a
   **classed object carrying its `key_policy` version and a schema version in
   class metadata**, with an internally **injective** length-prefixed/binary
   framing of its component tuple; a printable form exists for diagnostics only
   (S5 key-representation row, line 305; BLOCKER 1 corrections 2 and 6). It is
   **never** constructed by ambiguous delimiter concatenation — adversarial
   component values containing separators or control bytes must not collide
   (S5 injective-framing test, line 399). This is the **collision rule**: field
   distinctness is guaranteed by framing, not hoped for by escaping.

3. **Cleaner/display non-interference invariant (load-bearing).** For a fixed
   parse standard and key policy, changing **any** cleaning, profile, or
   display-only option — `path_encoding`, `host_encoding`, `protocol_handling`,
   case presentation, `www_handling`, subdomain/PSL trimming, every query
   cleaning dial, `port_handling` presentation, or a whole `profile` bundle —
   must leave the key **byte-identical** (S5 cleaner/profile-invariance row,
   line 331; display/cleaning-invariance test, line 400). A key changes **only**
   when an explicitly named comparison policy field changes. This is the
   executable form of the identity/presentation split and the direct remedy for
   the shipped defect where `path_encoding` forwarded through the join changed
   equality (S5 HIGH 3 minimal example, lines 186-188).

4. **Precedent already shipped.** The identity/presentation split is not novel to
   rurl: `domain_ascii` / `domain_unicode` are already **encoding-independent
   stable identity keys** that stay fixed regardless of `host_encoding`
   presentation (RURL-owrdsivt; `registers/public-surface-inventory.md` rows for
   `domain_ascii`/`domain_unicode`; ADR 0011's identity-vs-presentation axis).
   `get_url_key()` generalizes that same discipline to the whole URL.

### D-B. Versioned key policy, and the scheme/port truth tables

Equivalence is governed by a **single, symmetric, versioned `url_key_policy`
object** — never side-specific, because equality must stay symmetric and
transitive (S5 join key-policy row, line 373). Its version lives in the key's
class metadata; **a default or policy change bumps the key version and never
silently reinterprets already-persisted keys** (S5 stability/versioning row,
line 333). Component defaults below are the agent's recommended safe baseline;
the ones flagged are owner decisions (Open Questions).

**Scheme equality (enum, default `exact`).** Three named modes, exactly as the
protocol names them: `exact`, `http_https`, `http_https_missing` (S5 scheme-equality
row, line 309). `exact` means **same normalized scheme identity** (case-folded),
not original letter case; raw spelling is retained only for round-trip
(S5 scheme-case row, line 310). Recommended default is `exact` for join safety;
relaxed modes collapse **only** the named web-scheme states and leave every other
valid scheme exact. Original scheme *presence* (explicit vs inferred vs missing)
is preserved separately from the parsed effective scheme; inference is **never**
used solely to erase a "missing" marker (S5 scheme-source row, line 308).

**Port equivalence (default-port truth table).** Recognized default ports are
normalized per each row's own explicit scheme, **before** any optional scheme
collapsing; non-default ports are always key-significant; another scheme's
default is never stripped; a missing-scheme port stays literal and is never
labelled default (S5 port rows 311-313 and truth-table skeleton, lines 339-353):

| Left state | Right state | `exact` | `http_https` | `http_https_missing` |
|---|---|---:|---:|---:|
| HTTP absent port | HTTP `:80` | equal | equal | equal |
| HTTPS absent port | HTTPS `:443` | equal | equal | equal |
| HTTP absent port | HTTP `:8080` | distinct | distinct | distinct |
| HTTPS absent port | HTTPS `:80` | distinct | distinct | distinct |
| HTTP `:443` | HTTPS absent | distinct | **distinct** (recommended) | **distinct** (recommended) |
| HTTP `:80` | HTTPS `:443` | distinct (by scheme) | **distinct** (recommended) | **distinct** (recommended) |
| HTTP absent | HTTPS absent | distinct | equal | equal |
| Missing scheme/no port | HTTP/no port | distinct | distinct | equal |
| Missing scheme `:80` | HTTP/no port | distinct | distinct | **distinct** (recommended; no effective-scheme guessing) |
| Missing scheme `:443` | HTTPS/no port | distinct | distinct | **distinct** (recommended) |
| Scheme-relative/no port | missing host-shaped/no port | **distinct** (recommended) | distinct | distinct |
| FTP `:21` | FTP/no port | **exact literal** (recommended; HTTP/S-only equivalence in v1) | — | — |
| Custom scheme `:123` | custom/no port | distinct | distinct | distinct |

The rows the S5 skeleton marked `?` are **proposed `distinct`** here (no
mismatched-explicit-port collapse; no missing-scheme effective-port guessing) and
are raised as Open Questions 3-4 for the owner to confirm or override. Default
ports beyond HTTP/HTTPS (ftp/ws/wss/custom) are **not** normalized in comparison
v1 without an explicit versioned policy widening (S5 other-scheme-defaults row,
line 313).

**Component equivalence defaults (recommended baseline; owner-confirmable).**
Host: compare the internal normalized domain/IP identity, never Unicode/punycode
display; `www`/subdomain/PSL edits excluded from the default key (S5 rows
315-318). Path: compare the post-standard identity path before presentation,
with reserved bytes preserved — `%2F` never silently becomes a structural `/`
(S5 rows 319-322; the direct fix for HIGH 3). Query: preserve
absent/present-empty/present-nonempty structural state; recommended default is
exact structural query (an SEO-style ignore/filter is an explicit, separately
named policy, never a display dial) (S5 rows 323-328). **Fragment and userinfo:
recommended ignored for web-resource identity, with their structural state kept
outside the key for diagnostics — an explicit owner decision, never decided by
display stripping** (S5 rows 329-330; Open Question 5).

### D-C. Eligibility, invalid rows, and collisions

**Eligibility states** are enumerated and never conflated: OK, warning, invalid
parse, unsupported-but-valid scheme, relative/opaque reference, missing input,
empty input (S5 eligibility row, line 306). Non-keyable rows produce an **`NA`
key plus a companion diagnostic reason**; the public key vector uses `NA` *only*
for non-keyable rows. **Missing input is never conflated with invalid parse.**
Default match behavior for missing/non-keyable rows is **never-match** unless the
caller explicitly opts into missing-equality, and even then missing-input may
match only missing-input, never an invalid parse (S5 missing/invalid-key row,
line 332; join `NA`-matching row, line 378). **Collision** at the key level is
prevented structurally by the injective framing of D-A(2); collision at the
*join* level (a key appearing on multiple rows) is a multiplicity/relationship
question owned by D-D, **not** silently resolved by discarding rows.

### D-D. Six-join semantics keyed on the identity key

A named public family replaces the ambiguous "dplyr-shaped" language with a
pinned contract, all keyed on `get_url_key()` (S5 BLOCKER 2, lines 162-172;
join-semantics matrix, lines 359-366):

`url_inner_join`, `url_left_join`, `url_right_join`, `url_full_join`,
`url_semi_join`, `url_anti_join`.

Each pins: retained input rows, match multiplicity, a stable row-order algorithm,
output-column selection/order, and invalid/missing default — per the S5
join-semantics matrix (lines 359-366). Cross-cutting contract, from the S5
cross-cutting matrix (lines 370-392):

- **Relationship** validated on eligible keys **before** result materialization
  (enum: none/one-to-one/one-to-many/many-to-one/many-to-many); the error carries
  representative keys/counts without leaking credentials (S5 line 375).
- **Multiple matches** default `all`; any lossy narrowing is a separately named
  `multiple` option with defined stable first/last order — it does **not** reuse
  the shipped `collision` dial (S5 line 376; the fix for HIGH 7's dangerous
  `collision = "first"` silent row-discard).
- **Warnings vs invalid** are separated and renamed away from the misleading
  `on_parse_error` (`invalid = keep/drop/error` and `warnings = allow/reject/error`)
  (S5 line 379; C7/HIGH 7).
- **Key visibility** default hidden or a collision-proof explicit name; when
  exposed it is the **classed non-URL key, never a URL-looking `JoinKey`
  string** (S5 line 381).
- **Suffix / name repair, zero-row prototypes, stable order** are pinned per join,
  not left to incidental `merge()`/`data.frame()` behavior (S5 lines 383-388).
- **Resource guard:** preflight key counts / relationship before materializing a
  Cartesian product (S5 line 390).
- **Dependency meaning:** the family is pinned as an **independent rurl contract**
  (recommended) or against a named dplyr version — chosen explicitly, not implied
  by the phrase "dplyr-shaped" (S5 line 392; Open Question 6).

### D-E. C-06 disposition and the `canonical_join()` migration

**C-06 disposition: `MIGRATE` — the shipped `clean_url`-keyed join is LEGACY /
COMPATIBILITY-ONLY, explicitly NOT the v3 identity model, and is migrated on a
versioned, non-silent path.** This applies C-06's required disposition verbatim
(*"Introduce the independent key contract and an explicit migration plan; do not
reinterpret the shipped join as v3-compliant"*; §5 line 329) and RCON-07's
verbatim finding (*"Shipped `canonical_join()` derives its key from lossy
`clean_url`; that is legacy behavior, not the proposed v3 identity model"*; §4
lines 224-245). The shipped behavior is **not** relabelled compliant — it is
named legacy and given a migration.

Verified shipped anchor: `.cj_side_state()` assigns `key <- parsed$clean_url`
(`R/canonical_join.R:243`), the join is a base `merge(by = ".join_key",
sort = FALSE)` (`R/canonical_join.R:151-157`), and its four modes / `collision`
/ `on_parse_error` / `join_parse_status` args are the shipped signature
(`R/canonical_join.R:92-100`). §10 direct evidence records this as *"Current join
identity is `clean_url` — `R/canonical_join.R` key construction — Verified."*

Migration plan:

1. **Freeze the legacy behavior as an explicitly named policy.** `canonical_join()`
   is retained for a documented deprecation window, but its `clean_url` keying is
   surfaced as an **explicit legacy key policy**, not as implicit equality
   inherited through `...` (S5 HIGH 6 correction, line 252). The unrestricted
   `...` that currently makes every cleaning/display dial an equality dial (S5
   line 94) is closed: comparison-irrelevant cleaning/display arguments **warn**.
2. **Introduce the identity-keyed family alongside it.** The six D-D joins on
   `get_url_key()` are the v3 identity-keyed replacement; `canonical_join()`
   delegates to `url_inner_join()` (etc.) once a caller opts into the identity key
   policy.
3. **Provide a migration audit surface.** A dual-key comparison / audit helper
   lets users quantify how many matches change between the legacy `clean_url` key
   and the v3 identity key **before** switching (S5 HIGH 6 correction, line 252;
   Open Question 20 in S5). No caller is silently re-matched.
4. **Per-argument migration disposition.** Every current forwarded dial
   (`profile`, `url_standard`, `scheme_acceptance`, `scheme_policy`,
   `protocol_handling`, `www_handling`, host/path/query/port cleaning dials,
   `path_encoding`, `case_handling`) gets a row stating whether it is: a legitimate
   parse-policy input to the key, an explicit comparison policy, or a
   presentation dial that no longer touches equality (warned/rejected). This is
   the migration table S5 HIGH 6 requires (lines 234-252).

Back-compat boundary: within the deprecation window a legacy call keeps its
shipped result **only** under the explicit legacy policy; the v3 default for the
new family is the identity key. Removing `canonical_join()` or flipping its
default key is a later versioned step, not this record.

## Consequences

- **C-06 becomes decidable:** reviewers point to one predicate — the key is the
  post-standard identity record, and cleaning/display cannot change it — for
  "do these two URLs compare equal?".
- **A whole class of false joins is closed:** default-port collapse
  (`http://x` vs `http://x:8080`), display-driven `%2F`→`/` path collapse, and
  scheme-inference erasure of "missing" no longer silently match, because the key
  is built from identity state with an explicit truth table (D-B).
- **The equality surface becomes discoverable:** comparison policy lives in a
  named `url_key_policy`, not in an unrestricted `...` where every cleaning dial
  was secretly an equality dial (S5 line 94).
- **A versioned-key tripwire is established:** persisted keys are safe across
  releases because a policy/default change bumps the key version rather than
  silently reinterpreting stored keys (D-B).
- **Migration is auditable, not silent:** the dual-key audit helper lets users
  measure changed matches before switching (D-E.3); no caller is re-matched
  behind their back.
- **New maintenance surface:** the key-policy matrix, scheme/port truth table,
  and six-join matrix must stay in sync with the parser's identity record and the
  profile bundles; G3's contract matrices are the mechanical check.
- **Downstream G3** inherits concrete tables to make normative; the register
  single writer can close C-06 at G2.A citing this record.
