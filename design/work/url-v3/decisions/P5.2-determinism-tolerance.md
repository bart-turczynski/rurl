# P5.2 — Determinism tolerance and exception policy

## Decision (PROPOSED)

> This record is **proposed, not binding**. It is authored at `state: PROPOSED`
> under the P0.1 mechanism; only the owner's merge moves it to `ACCEPTED`
> (P0.1 §Decision4, §Ratification). Until then nothing here is normative, and
> `accepted_at` / `accepted_evidence` read `pending` because the ratifying act
> does not yet exist.

C-09 (reconciliation §5) requires that the determinism workflow — which today
**"calls itself evidence collection, never fails on divergence, and runs
automatically only for harness changes"** — be replaced for v3 by *"a separate
failing acceptance comparison and approved-exception mechanism."* This record
settles that disposition in three parts.

### 1. A failing determinism acceptance gate (not evidence collection)

v3 introduces a **determinism acceptance gate** that is a distinct mechanism
from the shipped `determinism-probe.yml`. The shipped probe is retained,
unchanged in posture, as non-gating research evidence: its header states
**"EVIDENCE COLLECTION, NOT A GATE … a divergence must NEVER fail the build"**
and only a probe that could not run at all fails
(`.github/workflows/determinism-probe.yml:1-6`). That property is deliberate for
a research sweep and is **not** edited by this record. The acceptance gate is
additive.

**What the gate compares.** The gate compares the **canonical parse dumps** — the
full-string plus decomposed-state output that `tools/determinism/parse-dump.R`
emits per cell as `dump-<LABEL>.csv` (at the frozen 2026-07-20 run: 1995 rows
per cell over the committed `tools/determinism/corpus.csv`, compared on the
`id` + `url_standard` key across the 20 non-key component/state columns; run
`29787127184`, `main` @ `1cf200f`, README "The charset axis, re-measured"). The
comparison unit is the **canonical dump projection** of each cell — the keyed,
column-normalized dump, not raw CSV bytes — reduced to one content hash per
cell, exactly as the frozen run already demonstrates ("all 17 `dump-<LABEL>.csv`
files are **byte-identical** — one MD5 `3db0d6e9…` across the whole sweep").

**Which cells are comparable.** The gate compares only **cross-platform /
cross-run comparable cells**: the OS × R-build × charset/locale matrix with the
`cross_os_comparable` flag true. Runner-default cells stay separately labeled
and **out of the equality comparison** exactly as today
(`determinism-probe.yml:108-114`, `locale-<LABEL>.csv` records
`cross_os_comparable=false`); their divergence from the same OS's `utf8` cell is
recorded as data, never gated. The comparable-cell **set is part of the gate
definition**, so a cell absent by construction (no upstream R build for that
tuple, e.g. `macos-latest`/`devel`) is a declared matrix exclusion, **not** a
divergence and **not** a failure. Per RCON-09 the comparable axis set is
required to extend beyond OS/R/locale/charset to **cache state**
(cold / warm / disabled) and **PSL engine snapshot** (explicit vs session
default) as those graduate; the cache-transparency contract that defines those
states is P5.1's, and the engine/default-snapshot provenance is P0.4/RCON-09
territory — this record owns only their inclusion in the equality gate.

**When it runs.** The gate runs on the change surfaces that can perturb parse
output — **not only harness changes** — namely: the R parser/serializer/encoding
sources (`R/**`), the dependency and namespace surface (`DESCRIPTION`,
`NAMESPACE`), the determinism harness and corpus (`tools/determinism/**`),
and pinned dependency bumps (`pslr`, `punycoder`). It additionally runs on a
schedule and as a **required pre-graduation gate for the curl-free 3.0 slice**
(reconciliation §7 G4; S8 acceptance gate 8). This closes the C-09 finding that
the probe "runs automatically only for harness changes" and the S8 minimal
example that "a parser change can alter Windows C-locale output; normal CI stays
green, and the evidence workflow neither runs nor fails automatically."

**Pass/fail rule.** The gate **FAILS the build** when any two comparable cells,
or two pinned runs of the same comparable cell, differ on the canonical dump
projection **and no matching approved exception (part 3) covers that
divergence**. A divergence covered by an unexpired, in-scope approved exception
**passes** (and is still recorded). Any **new or uncovered** divergence fails.
A comparable cell that was expected but **could not produce its canonical dump**
(a `DEGRADED-<LABEL>` outcome) is a **gate failure** — unlike evidence
collection, which tolerates a degraded cell — unless that gap is itself a
registered exception, because an un-runnable comparable cell cannot prove
determinism.

### 2. Determinism tolerance: zero unapproved divergence

The tolerance is **byte-exact zero-divergence on the canonical outputs**, across
all comparable cells and pinned runs. There is **no fuzzy numeric threshold**
and **no per-column slack budget**. This is the strictest defensible rule and it
is not aspirational: the frozen 2026-07-20 sweep already achieves it — all 17
comparable dumps reduce to a single MD5, and "the parse output no longer depends
on OS, R version, language or character set at all." Zero-divergence is
therefore the demonstrated baseline, so anything weaker would be a regression
dressed as a tolerance.

Named, bounded platform allowances (for example platform-specific floating-point
formatting, or locale collation surfacing in ordering) are **not** encoded as a
fuzzy threshold in the gate. Tolerance is expressed **only** through the explicit
approved-exception mechanism (part 3), each allowance being a *named, bounded,
owner-approved, expiring* register row — never an ambient "close enough" margin.
This mirrors P5.3's deviation-ledger discipline: a divergence is either exactly
zero, or it is a specific approved exception; there is no third, unnamed state.

### 3. Approved-exception mechanism

A **determinism-exceptions register** (an append-only row register under
`design/work/url-v3/registers/`, e.g. `determinism-exceptions.md`, in the P0.2
register form) records each *known, justified* divergence so the gate passes for
it while still failing on any **new** divergence. G4 implements the register and
the gate's consumption of it; this record fixes its contract. Each exception row
carries at minimum:

| Field | Meaning |
|---|---|
| `exception_id` | Stable identity for the allowed divergence. |
| `owner` / `approver` | The owner who authorized it (P0.1 authority). |
| `justification` | Why the divergence is acceptable and irreducible. |
| `scope` | The exact comparable cells / platforms / columns / rows it covers. |
| `signature` | The divergence fingerprint it authorizes (the specific diff), so it cannot silently widen to unrelated divergences. |
| `expiry` | A date or version bound; an expired exception stops matching and the gate fails again. |
| `tracking_issue` | The `fp` issue owning the remediation or permanent-acceptance decision. |
| `state` | Lifecycle state (`§6` lifecycle). |

**Matching rule.** A divergence is authorized **iff** its signature is within an
exception's `scope`, the exception is unexpired, and its `state` is active. Any
divergence with **no matching approved exception FAILS** the gate. This is the
"approved-exception mechanism" C-09 requires and it is the *only* channel by
which tolerance is granted.

## Rationale

- **C-09 verbatim disposition (reconciliation §5):** the required disposition is
  literally *"Add a separate failing acceptance comparison and approved-exception
  mechanism."* Parts 1 and 3 supply exactly those two pieces; the shipped probe
  is left as the separate research sweep it declares itself to be.
- **RCON-10 (§4):** conformance/determinism/migration/release claims "are not
  executable"; the current determinism workflow "is explicitly evidence
  collection and does not fail on cross-platform divergence." The required
  amendment names "a failing determinism comparison gate with approved
  exceptions." Part 1's pass/fail rule and part 3's register are that amendment.
- **RCON-09 (§4):** determinism of caches / PSL engine / parallel execution is
  "not measurable." Part 1 requires the comparable axis set to extend to cache
  state and engine snapshot, while deferring the *definitions* of those states to
  P5.1 (cache) and P0.4/RCON-09 (engine/default snapshot) — this record gates
  them without redefining them.
- **§7 G4:** "determinism … properties have positive and negative coverage" and
  every normative claim maps to tests or "a documented manual verification with
  exact commands." The zero-divergence tolerance (positive) plus the failing
  comparison and the DEGRADED-is-failure rule (negative) give both signs; G4
  implements the executable form.
- **§8 P5:** "determinism tolerance" is an enumerated P5 owner decision. Part 2
  answers it: zero unapproved divergence, tolerance only through the exception
  register.
- **§10 direct evidence:** the shipped posture ("EVIDENCE COLLECTION, NOT A
  GATE", narrow triggers, 17-dump structure) is verified against
  `.github/workflows/determinism-probe.yml` and
  `tools/determinism/README.md`; "Verified by explicit workflow comments and
  narrow triggers." The gate is defined as *additive to*, not an edit of, that
  probe.
- **S8 (primary evidence, `S8-performance…-migration.md`):** finding "High 7 —
  Determinism is evidence collection, not an acceptance gate" and its
  §"Determinism evidence and gate gap" require the gate to define "what
  bytes/fields are compared, which cells are comparable, what failure means,
  where the result manifest is retained, which changes trigger it, and which
  graduation step requires it." Parts 1–3 answer each: the canonical dump
  projection; comparable = `cross_os_comparable` cells; failure = build fail on
  uncovered divergence; retention via the probe's 90-day artifacts plus the
  register; triggers on parser/dep/harness changes plus pre-graduation.

## Consequences

- G4 must ship: (a) a failing comparison over the comparable dump projections;
  (b) the `determinism-exceptions` register and its matching engine; (c) triggers
  on `R/**`, `DESCRIPTION`, `NAMESPACE`, `tools/determinism/**`, and dependency
  bumps, plus schedule and pre-graduation; and (d) retention of the compared
  dumps and the pass/fail manifest.
- The curl-free 3.0 slice **cannot graduate** until the gate passes over the
  comparable matrix (S8 acceptance gate 8; reconciliation §7 G4). After curl removal the
  old libcurl-version Docker matrix becomes historical causality evidence only,
  and the gate's oracle is the curl-free parser/serializer output.
- Any platform divergence that cannot be eliminated must be **named, justified,
  scoped, and expired** in the register before the gate will pass — there is no
  silent tolerance and no green build over an unowned difference.
- Because zero-divergence is already demonstrated, the gate is expected to be
  **green with an empty exception register at acceptance**; the register exists
  to make *future* justified divergence explicit, not to paper over a current
  gap.

## Open questions for the owner

1. **Comparable axis set at acceptance.** Adopt the OS × R × charset/locale
   matrix now and require cache-state (cold/warm/off) and engine-snapshot cells
   to be **added as P5.1 / P0.4 land** (recommended), or require those axes in
   the gate's first accepted form? Recommendation: adopt OS/R/locale/charset now;
   gate the cache/engine axes on their defining records to avoid forward
   coupling.
2. **Trigger surface.** Confirm the change paths that fire the gate
   (`R/**`, `DESCRIPTION`, `NAMESPACE`, `tools/determinism/**`, dep bumps) plus a
   schedule and the pre-graduation requirement — or narrow/widen them.
3. **macOS R-devel exclusion.** Confirm the structural `macos-latest`/`devel`
   omission (no upstream build) stays a **matrix-definition exclusion**, not a
   register exception (recommended — it is an absence, not a divergence).
4. **Exception expiry policy.** A hard maximum window (e.g. one minor release)
   for every exception, or per-exception expiry with owner review at renewal?
   Recommendation: per-exception expiry, with no exception outliving the release
   line it was granted against.
5. **DEGRADED comparable cell.** Confirm a degraded comparable cell is a **hard
   gate failure** (recommended) rather than an auto-tolerated gap, so an
   un-runnable platform cannot masquerade as determinism.
6. **Canonical comparison projection.** Confirm the compared unit is the **keyed,
   column-normalized dump projection** (`id` + `url_standard` over the non-key
   columns) rather than raw CSV bytes, so incidental CSV formatting never
   triggers a false failure (recommended; matches the frozen run's method).
