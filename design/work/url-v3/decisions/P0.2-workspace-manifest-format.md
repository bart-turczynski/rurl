# P0.2 — Durable workspace and manifest format

## Decision

### 1. Canonical tracked workspace

The single tracked protocol workspace is **`design/work/url-v3/`** — already the
home of `protocol-review-reconciliation.md`. Its layout:

| Path | Contents | Mutability |
|---|---|---|
| `design/work/url-v3/manifest.yaml` | Section 6 artifact 1 (the manifest). | Single-writer (owner); versioned |
| `design/work/url-v3/decisions/` | P0-P5 owner-decision records (this format). | Append-only |
| `design/work/url-v3/registers/` | Decision, contradiction, finding, and source-claim registers. | Append-only rows |
| `design/work/url-v3/evidence/` | **Raw-evidence archive exception** — the frozen protocol, the DRAFT PRD, and the S1-S9 reports graduated out of gitignored `_scratch/`. | Read-only / immutable after graduation |
| `design/work/url-v3/protocol-review-reconciliation.md` | The accepted reconciliation (unchanged). | Superseded only by a new record |

`design/work/url-v3/` is preferred over a new top-level `v3/` or `design/v3/`
root so the workspace stays contiguous with the already-merged reconciliation
and the repository's `design/` home; no namespace gain justifies the split.

### 2. Raw-evidence archive exception

Eleven files are graduated **verbatim** into tracked
`design/work/url-v3/evidence/` (this is G0.1's payload): the nine S1-S9 reports
from gitignored `_scratch/orchestrate/v3-protocol-review/` (2,974 lines /
399,102 bytes, per reconciliation §2 and §60), plus the protocol
`url-v3-spec-reconstruction-protocol.md` (334 lines / 17,563 bytes) and the
DRAFT PRD `prd-url-object-grammar-DRAFT.md` (300 lines / 16,084 bytes) from the
parent `_scratch/` directory — **3,608 lines / 432,749 bytes** in total. They
are frozen inputs: once graduated they are read-only and are referenced by
stable identity and checksum, never edited. Their SHA-256 values are already
recorded in the reconciliation Section 2 and are re-pinned in the manifest.

### 3. Manifest format

The manifest is **YAML** (`manifest.yaml`) — human-diffable and
machine-readable, and the repository already parses YAML. Required keys:

```yaml
schema_version: "1.0.0"        # semver; this manifest schema
source_baseline: 89be90b        # immutable historical SHA; safe to embed
control_plane_snapshot: v3/cp-snapshot-1   # annotated tag resolving to the C1 content commit
repo_commit: v3/cp-snapshot-1              # same tag; the content commit this manifest describes
sources:                        # frozen source inventory
  - path: design/work/url-v3/evidence/S1-object-state-vector.md
    sha256: dfde4d9d...         # reuse the Section 2 hashes
    identity: S1
    frozen: true
artifacts:                      # hashes of the tracked control-plane artifacts
  - path: ...
    sha256: ...
tracker_snapshot:               # frozen fp export (canonical path + command fixed by P0.3)
  path: design/work/url-v3/evidence/tracker-snapshot.json
  sha256: ...
  as_of_commit: v3/cp-snapshot-1   # git tag, per the self-reference rule
external_revisions:             # pinned dependency provenance
  pslr: ...
  punycoder: ...
tool_versions: { ... }
restore_instructions: |         # fresh-clone restore drill (G0.4)
  ...
```

Field-level rules:

- **Stable identities.** Every source and artifact carries a stable `identity`
  (e.g. `S1`) independent of its path, so a later move does not break trace
  links.
- **Checksums.** SHA-256 for every frozen source and every tracked artifact,
  reusing the already-published Section 2 hashes where they exist.
- **Schema versioning.** `schema_version` is semver; a breaking change to the
  manifest shape bumps the major and is itself a superseding P0-tier record.
- **Baseline vs snapshot are distinct.** `source_baseline` (what the reviews
  analyzed, `89be90b`) and `control_plane_snapshot` (when durability was
  established) are always separate keys and are never conflated.
- **Self-reference rule (no forward-hash paradox).** A manifest cannot embed the
  hash of the commit that contains it, nor its own content hash. Both are
  resolved by a **two-phase seal**:
  1. The `artifacts` hash list **excludes `manifest.yaml` itself**; the
     manifest's own integrity is supplied by git's object hash of the commit
     that contains it, not by a self-entry.
  2. `repo_commit`, `control_plane_snapshot`, and `tracker_snapshot.as_of_commit`
     are **not written as raw SHAs inside the manifest** (a commit cannot embed
     its own hash). The snapshot is sealed in this explicit order:
     - **a. Land C1 (content commit).** Merge the completed manifest and evidence
       to `main`; **C1 is that merged content commit** and now has a durable
       `main` SHA. Its three snapshot keys hold the tag name `v3/cp-snapshot-<n>`
       (they cannot hold C1's own SHA — hence the tag indirection).
     - **b. Create the tag.** As a *separate ref operation* (a tag is a ref, not
       a commit, and is not "created by" any commit), create and push an
       **annotated** tag `v3/cp-snapshot-<n>` pointing at **C1**.
     - **c. Bind in the log (seal PR).** A separate seal PR appends one line to
       the append-only register `design/work/url-v3/registers/snapshots.log`:
       `<n>  v3/cp-snapshot-<n>  <C1 full SHA>  <UTC timestamp>`, recording C1's
       now-known SHA as the durable pinned revision.
     Because the binding records C1's already-known `main` SHA, the pinned
     revision is a real, durable, git-tracked value — not merely a movable ref.
  3. **Snapshot immutability.** A `v3/cp-snapshot-<n>` tag, once created, is
     **never deleted or re-pointed**; a corrected or later snapshot takes the
     next integer `<n>`. The `snapshots.log` line is the authority if a tag is
     ever tampered with; restore and the G0.4 drill verify tag→SHA against that
     log. This mirrors the P0.1 §Ratification seal and removes every
     forward-reference to an unknowable hash.
