# P0.3 — Tracker topology, snapshot boundary, mutation and single-writer rules

## Decision

### 1. Final epic placement

Epic **`RURL-dorofzmb`** is confirmed as a **top-level root epic** — its current
placement. It is deliberately *not* nested under the determinism epic
`RURL-gxqdmpcp` (which parented the reconciliation `RURL-jairsufw`), because
protocol hardening is a distinct program with its own gate sequence. This
decision ends the "placement provisional until P0.3" condition. Placement is
frozen before the snapshot; any later reparenting requires a superseding
P0-tier record.

### 2. Snapshot boundary / as-of rule

Authority is read from a **frozen `fp` tracker export**, not from live tracker
state:

- **Canonical path (single).** The export is stored at exactly
  `design/work/url-v3/evidence/tracker-snapshot.json` — under `evidence/`
  because it is a frozen raw input, not a maintained register. There is no
  alternate location.
- **Exact reproducible command.** The `fp` CLI exposes no generic `export`; the
  snapshot is produced by its JSON output and is regenerable byte-for-byte from
  the frozen tracker state:

  ```sh
  fp tree --format json > design/work/url-v3/evidence/tracker-snapshot.json
  # companion flat listing, if a per-issue view is needed:
  fp issue list --format json > design/work/url-v3/evidence/tracker-issues.json
  ```

  Completeness rule: the export must contain the full epic subtree rooted at
  `RURL-dorofzmb` (every descendant issue, status, parent edge, and comment
  reachable from it). The file is committed and its SHA-256 recorded in the
  manifest.
- Every normative reference to "the tracker" means that frozen export as of the
  snapshot commit. Live tracker drift after the snapshot has no authority.
- The snapshot commit is the `control_plane_snapshot` in the manifest (named by
  the `v3/cp-snapshot-<n>` tag, per P0.2) and is kept distinct from
  `source_baseline` (89be90b).

### 3. Mutations permitted during review

- `fp` status transitions and comments remain **allowed** during the effort but
  are **explicitly non-authoritative** (P0.1 §5). They are working notes.
- Topology (parent links / reparenting) is **frozen after the snapshot**;
  changing it requires a new decision record, so every structural change is
  auditable.
- Issue creation/refinement for still-deferred leaves is allowed pre-snapshot;
  after the snapshot, new leaves are recorded against the frozen export delta.

### 4. Tracker-to-record ID rule

`fp` IDs (`RURL-...`) are **pointers, not authority**. Normative identity lives
in the tracked records: decision IDs (`P0.1`, `P2.3`, ...) and the register IDs
(`RCON-*`, `C-*`, and the forthcoming `REQ-*`). Each decision record names its
originating tracker issue in the `tracker:` field for recovery, but the record —
not the issue — is the authority. A tracker comment never becomes normative
prose.

### 5. Single-writer ownership

One writer owns each mutable control artifact:

| Artifact | Author (may propose) | Accepts / single writer |
|---|---|---|
| `decisions/*` (owner decisions) | agent (`PROPOSED`) | **owner** (merge = `ACCEPTED`) |
| `manifest.yaml` | agent | **owner** (merge) |
| `registers/*` | agent (rows at `DISCOVERED`/`PROPOSED`) | **owner** for `ACCEPTED` transitions |
| `evidence/*` | — (frozen at graduation) | immutable |

This is single-*writer*, not merely single-*acceptor*. For each mutable
artifact:

- The **owner** is the sole writer of its *accepted* state (only an owner merge
  moves it to `ACCEPTED`; P0.1 §4).
- *Proposal* authorship is **serialized**, not concurrent: at most one open PR
  may touch a given artifact at a time. A second proposer must wait for that PR
  to merge or close and then rebase, so no two branches hold divergent uncommitted
  state for the same artifact. This gives each artifact exactly one writer of
  record at any instant, satisfying G0's "one writer owns each mutable artifact".
- Registers are the one nuance: rows are append-only, so distinct proposers may
  add *different* rows concurrently, but no two may edit the *same* row; a row's
  writer of record is its author until an owner-accepted transition supersedes
  it.

  freeze; the export hash lands in the manifest.
- **G0.4** (restore drill) reconstructs authority from the frozen export plus
  the manifest, not from live tracker state.

  editing the tree.

  is ratified *under* P0.1 and *consumes* P0.2's canonical paths and manifest
  keys. Per §4 the record's `depends_on` is authoritative; the `fp` dependency
  edge is a non-authoritative pointer and is aligned to match, but if the two
  ever diverge the record governs.
